## Root Cause

The repro (from timm dm_nfnet_f0 train) computes an adaptive-average-pool backward fused with GELU derivative and channel sum:

1. Input: [128, 3072] pooled gradient `mm`
2. Reshape -> as_strided_scatter -> as_strided [128,3072,1,1] -> expand [128,3072,6,6] -> div by 36
3. Multiply by GELU derivative of `convolution_80` [128,3072,6,6]
4. Sum over dims [0,2,3] -> output [3072]

The key insight: the `as_strided_scatter` + `expand` + `div` is a structured pattern that broadcasts [128,3072] into [128,3072,6,6] with each spatial position getting `mm[n,c]/36`. This is functionally just `mm[n,c] * (1/36)` broadcast over HW.

The oracle recognizes this structure: instead of materializing the full [128,3072,6,6] expanded tensor, it directly computes `mm[n,c]/36 * gelu_grad(conv[n,c,h,w])` and reduces to [3072] in a single kernel with atomic channel accumulation.

Inductor:
1. Kernel 1: materializes the full [128,3072,6,6] pointwise (expand+div+gelu_grad+mul)
2. Kernel 2: reduces the [128,3072,6,6] result to [3072]

The oracle avoids materializing the intermediate [128,3072,6,6] buffer (108 MB at fp32).

## Fix Assessment
**Design doc** - Requires a new FX pass or scheduler enhancement.

### What's needed:
An algebraic rewrite pass that recognizes `as_strided_scatter -> as_strided -> expand -> div -> pointwise -> sum` as a "broadcast-multiply-reduce" pattern where the broadcast source can be folded directly into the reduction without materializing the expanded intermediate.

Alternatively, the scheduler could detect that:
- Producer A: [128, 3072] broadcast to [128, 3072, 6, 6]
- Consumer: reduction over [0, 2, 3] back to [3072]
- The broadcast dimensions are exactly the reduction dimensions
- Therefore: fuse the broadcast source directly into the channel reduction

### Relevant files:
- `/tmp/pytorch-work/torch/_inductor/fx_passes/post_grad.py` - could add a pattern matching pass
- `/tmp/pytorch-work/torch/_inductor/lowering.py` - as_strided_scatter lowering
- `/tmp/pytorch-work/torch/_inductor/scheduler.py` - broadcast+reduce fusion

### Difficulty: Medium
The algebraic rewrite is clean: when expand dims == reduce dims, the expansion can be eliminated by scaling the reduction accumulator. However, the `as_strided_scatter` pattern makes it harder to detect programmatically.
