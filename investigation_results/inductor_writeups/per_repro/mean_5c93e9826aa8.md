- **Model**: timm_adv_inception_v3 (inference)
- **Pattern**: 6-branch BN-inference -> ReLU -> cat([branch0, cat(1,2), cat(3,4), branch5]) -> spatial mean [128,2048,8,8] -> view [128,2048]
- **Ratio**: 1.447x (oracle 49.98us vs compile 72.32us)
- **Classification**: SCHEDULER_FUSION
- **Status**: Actionable

## Benchmark Results
- Oracle: 49.98 us
- Compiled: 72.32 us
- Ratio: 1.447x

## Root Cause

The oracle avoids materializing the concatenated [128,2048,8,8] intermediate tensor (128*2048*64*4 bytes = 64MB) by computing BN+ReLU+spatial-mean directly from each branch tensor into its corresponding channel range in the [128,2048] output. It:
1. Precomputes per-channel scale/shift for all 2048 channels across all 6 branches in one small kernel
2. Launches 6 separate branch kernels that each read from the original branch tensor (320/384/384/384/384/192 channels), apply the pre-computed affine + ReLU, and write the spatial mean directly to the correct output channel range

Inductor generates 2 kernels:
1. `triton_poi_fused_0`: A pointwise kernel that computes BN+ReLU for all 6 branches and writes the concatenated [128,2048,8,8] result (materializing 64MB)
2. `triton_per_fused_mean_1`: A reduction kernel that reads the 64MB buffer and computes the spatial mean to get [128,2048]

The 64MB intermediate buffer materialization is the primary source of the performance gap. Writing and then reading 64MB of data that could be reduced in-place wastes significant memory bandwidth.

## Kernel Count
- **Inductor**: 2 kernels (pointwise cat+BN+ReLU, then separate mean reduction)
- **Oracle**: 7 kernels (1 affine precompute + 6 per-branch reductions)

## Config Exploration
- `combo_kernels=True` + `coordinate_descent_tuning=True` already enabled
- The issue is that Inductor cannot fuse across the `aten.cat` into the downstream `aten.mean` - the cat forces materialization

## Design Doc

### Why Inductor Cannot Do This Today

The scheduler cannot inline `aten.cat` producers into a downstream reduction consumer. When multiple BN+ReLU branches are concatenated along the channel dimension and then reduced (spatial mean), the scheduler must:
1. Materialize the cat result (because the reduction consumer needs a contiguous view of all channels)
2. Launch the reduction separately

The oracle sidesteps this by never forming the concatenated tensor - each branch writes directly to its output channel range after reducing in-place.

### Enhancement Needed

**SCHEDULER_FUSION**: Teach the scheduler to recognize the pattern:
```
cat([branch_0, branch_1, ..., branch_N], dim=C) -> mean(dim=[H,W])
```

When the only consumer of the cat is a reduction over non-cat dimensions (here: spatial H,W), the cat can be "virtualized" - each branch can independently reduce its spatial dimensions and write to the correct offset in the output. This eliminates the intermediate buffer entirely.

The key insight is: **spatial mean commutes with channel concatenation**. If you mean-reduce each branch separately and then conceptually "concatenate" the results, you get the same answer as concatenating first and then reducing.
