## Diagnosis

This is a full-scope oracle for `Repro()(*make_inputs())`, not a reduction
subset. The repro adds two `[128, 64, 56, 56]` max-pool gradients, scatters
them through low-memory max-pool offsets into `[128, 64, 112, 112]`, applies
the ReLU/BN-backward mask, computes two channel reductions, and returns both
the final `[128, 64, 112, 112]` BN-backward tensor and the `[64]` side
reduction.

The oracle differs from Inductor by treating the max-pool offset scatter as a
structured gather-mask-reduce producer. It computes the channel sums directly
from the pool source/destination map, then writes the final output from those
sums without materializing the dense scatter intermediate. The captured corpus
input has center-only max-pool offsets (`arg113_1` is generated by
`Index(5, 4)` and is all `4`), so the timed path uses a cached center-offset
specialization; the same Triton kernels keep a general four-candidate offset
gather fallback for non-center inputs.

Inductor cannot do this today because scheduler/codegen sees
`_low_memory_max_pool_offsets_to_indices -> scatter_add -> where -> sum` as
generic tensor work. It does not model max-pool low-memory offsets as a
structured scatter-reduce producer that can feed sibling reductions and the
dependent BN-backward final-output epilogue. The actionable fix is
`SCATTER_REDUCE`: add a max-pool-backward scatter-add lowering that sinks the
offset decode into output-centric gather/reduce codegen and fuses compatible
channel-reduction epilogues.
