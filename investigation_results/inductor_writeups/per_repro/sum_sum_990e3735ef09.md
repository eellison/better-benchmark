## Scope

The diagnostic oracle covers the full `repro.py` computation. It consumes the
same six inputs: `mm_2 [4096, 4096]`, `arg12_1 [4096]`, `arg114_1 [8, 512,
4096]`, `arg124_1 [8, 512, 1]`, and the two shape parameters. It returns the
same single `float32 [4096, 4096]` tensor with stride `(1, 4096)`.

The Triton kernel computes both row reductions,
`sum(mm_2 * arg12_1)` and `sum(mm_2 * arg12_1 * arg114_1)`, in one
multi-accumulator row program and then writes the dependent transposed epilogue:
`arg124_1 * ((mm_2 * arg12_1) * 4096 - sum_x - arg114_1 * sum_x_arg114)`.

## Diagnosis

The oracle differs from Inductor by forcing the sibling reductions and the full
dependent epilogue into one row-owned Triton schedule with no intermediate
reduction tensors. A split-reduction diagnostic variant was also measured and
was slower (`104-108 us` range), so adding more parallel reduction structure did
not expose a faster floor for this shape.

Inductor cannot generally prefer this exact row-owned schedule because the
single program over 4096 columns has high vector/register pressure, while split
forms reread the large inputs and add launches. The historical best compile
already beats the handwritten full-scope oracle, so this repro should be treated
as bandwidth/schedule-cost limited rather than an actionable missing fusion.
The only useful Inductor change implied here is a profitability guard/cost model
that avoids promoting similar one-row multi-output schedules unless they beat the
tuned reduction template for the concrete shape.
