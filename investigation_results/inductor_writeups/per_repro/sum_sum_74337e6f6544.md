## Root Cause

The oracle computes the Longformer loss-gradient update (cross-entropy backward with ignore_index), both padded materialized outputs, and the sibling vocabulary column sum by replacing the dense label-vocabulary one-hot construction plus row reduction with a structured valid-label update and sparse label correction.

Inductor lowers the broadcast equality mask (iota == label), one-hot `where`, row sum, dense exp/update, padding, transpose, and column sum as generic scheduled work over the full `[8192, 50265]` domain. The one-hot mask is dense (410M elements) but only 8192 are non-zero.

## Kernel Count
- **Oracle**: Structured sparse kernels that avoid dense one-hot materialization
- **Inductor**: Multiple generic kernels over full [8192, 50265] domain

## Config Exploration
combo_kernels and coordinate_descent_tuning reduce time but cannot eliminate the fundamental dense-vs-sparse algorithmic gap. The problem is 412M elements in the one-hot mask vs 8192 active labels.

## Design Doc

The fix requires:
1. Recognition of the `iota == label` one-hot pattern
2. Decomposition into: dense softmax-backward base + sparse label correction
3. The row reduction `sum(one_hot_where, dim=1)` becomes a simple gather from the label positions
4. The dense exp/sub/add update can proceed without the one-hot intermediate
5. Column sum and padded stores can be fused with the base update kernel

This is an algebraic elimination problem: the one-hot is mathematically equivalent to a sparse update but Inductor treats it as a dense tensor operation. Status: design documented, awaiting implementation of the one-hot algebraic elimination pass.
