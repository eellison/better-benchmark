## Pattern

MT5 inference relative-position attention softmax:

- Scores input: `bmm_2`, `f32[192, 128, 128]`, contiguous.
- Relative-position bias table: `arg6_1`, `f32[32, 6]`, contiguous.
- Repro scope: view scores to `[32, 6, 128, 128]`, build the MT5
  bidirectional relative-position buckets from query/key iotas, embed the
  `[32, 6]` bias table, permute/unsqueeze it to `[1, 6, 128, 128]`, add the
  tautological decoder-position mask, add scores, compute the `any(eq(-inf))`
  all-masked-row guard, perform stable softmax over the last dimension, zero
  all-`-inf` rows, expand to `[32, 6, 128, 128]`, and return the final
  contiguous `[192, 128, 128]` view.

The structured mask generated by this captured shape is always true
(`iota(128) >= 0`), so the full-scope oracle preserves it as an algebraic zero
inside the fused computation rather than materializing a `0/-inf` tensor.

## Diagnosis

The oracle computes the full `Repro.forward` result in one Triton head-tiled
row-softmax kernel. It recomputes the cheap MT5 bucket predicates at point of
use, loads the relative bias table directly, folds the tautological mask, keeps
the all-`-inf` row guard, and writes the final contiguous output layout. This
avoids materializing the bucket tensor, permuted embedding, boolean guard, and
intermediate softmax tensors.

Inductor currently lowers the decomposed
`iota/log/embedding/permute/add/eq/any/amax/exp/sum/div/where` graph as generic
producer and online-softmax fragments. The missing optimization is a
relative-position attention-softmax template that fuses bucket lookup,
structured mask handling, all-`-inf` row handling, and the layout-only epilogue
into the row kernel.
