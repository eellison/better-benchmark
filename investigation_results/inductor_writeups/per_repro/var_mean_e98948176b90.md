## Diagnosis

The oracle computes Swin window-reverse (mapping window-partitioned [128*4*49, 512] rows back to spatial [128*196, 512] layout via index arithmetic), residual add, hidden-size-512 population LayerNorm, and affine epilogue in one autotuned Triton kernel with block_m=[1,2,4,8] row tiles. It computes the window-to-spatial index mapping inline (window_h, window_w, inner_h, inner_w -> addmm_row) and loads the projected tensor via scatter-gather directly in the reduction body.

Inductor emits 1 kernel but with suboptimal reduction strategy -- it uses Welford statistics with a second pass for the affine epilogue that re-reads the input, whereas the oracle keeps the row tile live in registers through both the statistics and affine passes.

## Root cause

Even though Inductor emits 1 kernel, its reduction codegen for the Swin window-reverse + residual + LayerNorm pattern uses Welford online statistics with a second pass to apply the affine transformation, requiring a re-read of the row data. The oracle avoids this by:
1. Loading the full row into registers once (512 elements fits in registers with block_m tiling)
2. Computing mean/variance from the register-resident data
3. Applying the affine transformation to the same register-resident data

The persistent looped reduction (`multi_kernel=3`) partially addresses this by keeping data resident.
