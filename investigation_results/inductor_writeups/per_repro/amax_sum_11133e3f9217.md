## Diagnosis

The oracle computes the complete GPT-Neo masked attention softmax in one Triton row kernel. It inlines the structured mask generation (arange predicates, cumsum segment equality, broadcasted bool attention mask), score masking, stable row softmax (amax/sub/exp/sum/div), and final [512, 128, 128] output layout in a single pass.

Inductor lowers the decomposed graph (slice/iota/index/where/add/amax/sub/exp/sum/div) as two kernels: one for structured mask construction (iota/index), and one for the masked softmax reduction.

## Root cause

Inductor materializes the structured mask (arange predicates + cumsum equality) into a separate pointwise kernel that writes an intermediate buffer. The oracle recomputes these cheap structured predicates inside the row-softmax kernel, avoiding the intermediate write and extra kernel launch.

## Kernel count
- Oracle: 1 kernel (fused mask generation + masked softmax)
- Inductor: 2 kernels (mask construction pointwise, then fused masked softmax reduction)

## Recommendation

Requires teaching Inductor's scheduler to inline cheap structured-mask producers (iota/cumsum-index predicates) directly into the softmax reduction consumer, avoiding materialization of the mask intermediate. This is a `STRUCTURED_MASK_INLINE_SOFTMAX` pattern -- the mask is computationally cheap (index arithmetic) and should be recomputed rather than stored.
