# sum_sum_876992e24e09

## Compile: 18.34us, Oracle: 14.43us, Gap: 1.27x

## Diagnosis: COOPERATIVE_SPLIT_K

## Root cause: Inductor schedules the Qwen/Llama RMSNorm-backward as 3 separate Triton kernels: a row-reduction kernel for the hidden-dimension dot products, a column-reduction kernel for the [1024] weight gradient, and a pointwise kernel for the transposed bf16 [1024,2048] output. The oracle fuses all three into a single row-tiled cooperative kernel that keeps bf16-rounded row-local values live while also materializing the layout-changing side output and finalizing column partials in one pass.

## Fix path: Add a row-tiled multi-output RMSNorm-backward reduction lowering that fuses the bf16 producers, row-summary epilogue, transposed stores, and channel-gradient finalization. This requires Inductor's scheduler/codegen to support cooperative split-K multi-output templates that keep bf16-rounded row-local values live while materializing a layout-changing side output.

## Status: design_todo

## Details

- Model: vllm_Qwen_Qwen3-0.6B (inference)
- Pattern: sum+sum reduction (RMSNorm backward)
- Inductor kernels: 3 unique Triton kernels
- Ops: view, add, mul, convert_element_type (bf16<->f32), sum([2],keepdim), sum([0,1],keepdim), pow, div, expand, permute
- Shapes: [2048,1024] bf16 inputs, [1024] weight gradient output, [1024,2048] transposed output
- The gap (1.27x) is moderate because the [2048,1024] tensor is relatively small (4MB in bf16), so kernel launch overhead of 3 kernels vs 1 is the primary driver
- The oracle achieves savings through: (1) single-pass row tiling that avoids rematerializing the f32 intermediate, (2) fused transposed store avoiding a separate permute kernel
- No existing Inductor config flag addresses this gap
