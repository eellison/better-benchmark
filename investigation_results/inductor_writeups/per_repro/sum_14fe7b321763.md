# sum_14fe7b321763 — SCATTER_REDUCE

## Summary
- **Model**: torchbench_vgg16_train
- **Pattern**: maxpool backward scatter_add + where mask + channel sum
- **Ratio**: 1.113x
- **Status**: Moderate gap — scatter-reduce fusion opportunity

## Operation
1. Takes `[128, 256, 28, 28]` f32 pooled gradient + `[128, 256, 28, 28]` i8 max indices + `[128, 256, 56, 56]` bool mask
2. Computes `_low_memory_max_pool_offsets_to_indices` on the i8 offsets
3. Scatters gradients via `scatter_add` into `[32768, 3136]` (= `[128*256, 56*56]`)
4. Views as `[128, 256, 56, 56]`, applies `where(mask, full_scalar, scattered)`
5. Reduces via `sum(dim=[0, 2, 3])` to produce `[256]` channel sum

## Kernel Count
- Inductor: 3 kernels (full init + scatter + reduction)
- Oracle: fused gather-mask-reduce

## Benchmark
- Oracle: 254.78 us
- Compiled: 283.58 us
- Ratio: 1.113x

## Root Cause Analysis

Inductor generates 3 kernels:
1. **triton_poi_0**: Initializes the `[32768, 3136]` full tensor with zeros
2. **triton_poi_1**: Computes max pool offset-to-index conversion and performs scatter_add
3. **triton_red_2**: Reads the scattered result, applies `where` mask, and reduces to `[256]`

The oracle instead works from the **input space** (the `[128, 256, 28, 28]` gradient) and, for each input cell, computes which output cell it contributes to. This avoids materializing the full `[32768, 3136]` intermediate (= 412 MB of f32) and instead accumulates directly.

**Key inefficiency**: The scatter_add must materialize a large intermediate `[32768, 3136]` f32 tensor (approximately 400 MB) that is immediately consumed by the reduction. The oracle avoids this by reversing the computation direction — computing from source cells and accumulating into the `[256]` channel bins directly.

## Why Inductor Cannot Do This Today

1. **scatter_add is opaque to fusion**: The scheduler treats `scatter_add` as an atomic-capable extern/lowering that cannot be fused with its consumer reduction. The reduction must wait for the full scatter to complete.

2. **No scatter-reduce pattern recognition**: Inductor's `fx_passes/` do not recognize the pattern `full(0) -> scatter_add -> view -> sum(dims)` as a structured reduction that can be computed from the source space without materializing the scatter output.

3. **Maxpool backward is not recognized as structured**: The `_low_memory_max_pool_offsets_to_indices` -> scatter pattern is a well-known maxpool backward that maps each input cell to exactly one output cell. This 1:1 mapping means the reduction can be done as a direct gather-reduce.

## Fix Direction

**FX pass (post_grad.py)**: Recognize `scatter_add(zeros, indices, src) -> sum(dims)` and rewrite as a direct source-space reduction with index-based binning:
- File: `/tmp/pytorch-work/torch/_inductor/fx_passes/post_grad.py`
- Pattern: When a scatter_add output is only consumed by a sum over dimensions that include the scatter dimension, the entire chain can be replaced by a grouped reduction from the source tensor.

**Scheduler enhancement**: Alternatively, the scheduler could recognize that scatter_add followed by reduction over the scattered dimension is equivalent to a source-space reduction, avoiding the intermediate materialization.

## Design Doc

The scatter-reduce fusion would:
1. Detect `scatter_add(full(0), indices, values).sum(scatter_dim_and_others)` 
2. Rewrite as: for each source element, add `values[i] * mask[scatter_target[i]]` to the appropriate output bin
3. This eliminates the O(output_size) intermediate in favor of O(input_size) work

For the maxpool case specifically, since each source cell maps to exactly one target, this is a simple grouped reduction.
