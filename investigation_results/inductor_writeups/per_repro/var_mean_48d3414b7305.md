# var_mean_48d3414b7305

## Compile: 11.81us, Oracle: 10.34us, Gap: 1.142x

## Diagnosis: SCHEDULER_FUSION

## Root cause: The oracle fuses grouped LayerNorm (GroupNorm with groups=32, group_size=2*256=512) with affine, ReLU activation, and padded 3x3 stride-2 low-memory maxpool-with-offsets into a single Triton kernel that produces both f32 pooled values and int8 offset indices. Inductor emits 2 kernels:
1. `triton_per_fused_add_mul_relu_rsqrt_sub_unsqueeze_var_mean_view_0` -- GroupNorm + affine + ReLU (materializes full [64,64,16,16] activation)
2. `triton_poi_fused__low_memory_max_pool_with_offsets_...` -- maxpool reading the materialized activation

The oracle eliminates the intermediate [64,64,16,16] f32 buffer materialization by computing GroupNorm+ReLU inline during the pooling stencil traversal.

## Kernel count
- Inductor: 2 kernels (norm+relu, then maxpool)
- Oracle: 1 kernel (fused norm+relu+maxpool)

## Config exploration results
- multi_kernel=0: 46.45us, multi_kernel=1: 51.13us, multi_kernel=2: 49.84us, multi_kernel=3: 50.25us
- No config helps; multi_kernel makes it worse
- coord_descent_tuning already enabled

## Fix path: This requires a SCHEDULER_FUSION enhancement. The scheduler needs to recognize that:
1. The GroupNorm+ReLU producer feeds exclusively into a stencil-based consumer (maxpool)
2. The intermediate [64,64,16,16] buffer is read-only-once by the maxpool
3. The maxpool's 3x3 stencil with stride=2 means each output pixel reads from a small neighborhood

The fix would need to sink the reduction-backed normalized activation into the maxpool loop nest. This is non-trivial because the maxpool consumer uses shifted index patterns (stencil), but the GroupNorm producer is row-local per group. The scheduler would need to prove that each maxpool output tile's input footprint maps to complete norm groups.
