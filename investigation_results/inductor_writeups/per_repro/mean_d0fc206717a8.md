# mean_d0fc206717a8 - Multi BN-ReLU Spatial Mean (Inception V3)

## Classification: SCHEDULER_FUSION
## True Floor: Yes
## Actionable: Yes

## Benchmark Results
- Oracle: 37.92 us
- Compiled: 68.54 us
- Ratio: 1.808x

## Root Cause

The oracle fuses six BN-inference-affine + ReLU branches, two 768-channel inner cats, one 2048-channel outer cat, and an 8x8 spatial mean reduction into a single logical operation that writes directly to the final [128, 2048] output. It does this by:
1. Precomputing affine parameters (scale/shift) from all 6 branches into a unified [2, 2048] buffer
2. Launching per-branch reduction kernels that read each branch's NCHW input, apply the precomputed affine + ReLU, and reduce HW=64 spatially, writing directly into the correct slice of the output

Inductor produces 2 kernels:
1. A pointwise kernel (`triton_poi_fused_0`) that materializes all 6 BN-affine + ReLU branches and performs the cat operations into a [128, 2048, 8, 8] intermediate
2. A persistent reduction kernel (`triton_per_fused_mean_1`) that reads the materialized 2048-channel cat and computes the spatial mean

The gap is the materialized intermediate: writing and re-reading [128, 2048, 8, 8] = 128*2048*64*4 = ~67 MB of data. The oracle avoids this by reducing spatially within each branch kernel, only writing the final [128, 2048] = ~1 MB.

## Kernel Count
- Inductor: 2 kernels (pointwise + reduction)
- Oracle: 7 kernels (1 affine precompute + 6 branch reduction kernels)

## Config Exploration
- `combo_kernels=True`: Already enabled, does not help (combo fuses the BN pointwise ops together but cannot sink the reduction)
- `multi_kernel=3`: Actually makes things worse (261.6 us) because it produces variant kernels but doesn't address the fundamental materialization

## Why Inductor Cannot Do This Today

The scheduler treats `aten.cat` as a materialization barrier. When the cat's consumer is a reduction (mean.dim), the scheduler cannot "look through" the cat to fuse each branch's pointwise producer directly into the reduction.

Specifically:
- `torch/_inductor/scheduler.py` lines ~336+: `can_fuse` checks whether nodes share memory dependencies, but the cat creates a hard boundary between the BN producers and the mean consumer
- `torch/_inductor/ir.py` lines ~10144+: `should_realize_on_reuse` forces the cat to materialize because it has multiple input sources

## Required Enhancement (Design Doc)

**Virtual Cat Through Reduction**: The scheduler needs to recognize the pattern:
```
[branch1_pointwise, branch2_pointwise, ...] -> cat(dim=C) -> mean(dim=HW)
```
and transform it into per-branch reductions that write into channel slices of the output. This requires:

1. Detecting fixed-shape channel cats whose sole consumer is a spatial reduction
2. Representing the cat as a virtual multi-source layout (no materialization)
3. Emitting per-branch reduction kernels that each know their output channel offset
4. This is a scheduler-level optimization in `scheduler.py` (not a codegen template)

**Affected repro count**: This pattern appears in many CNN architectures with multi-branch modules (Inception, GhostNet, EfficientNet). Related repros in this chunk: mean_f7170c220032.
