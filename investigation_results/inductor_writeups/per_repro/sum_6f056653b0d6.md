## Root Cause

The repro (from timm dm_nfnet_f0 training) computes the GELU-backward channel-sum over shape [128, 128, 48, 48]:
- 128 output channels
- Per-channel reduction over N*H*W = 128*48*48 = 294,912 elements
- Pointwise chain: mul, erf, exp, add, mul (GELU backward derivative)
- Final sum per channel -> f32[128] output

### Oracle approach (cooperative split-K):
The oracle splits the 294,912-element per-channel reduction into 16 cooperative partials (each handling ~18,432 elements), with a 128x16 grid. Each partial accumulates the GELU-backward pointwise chain into a float32 accumulator. A second small kernel reduces the 16 partials per channel to produce the final f32[128] output.

### Inductor approach:
Inductor uses a generic split reduction with fewer spatial partitions and a separate partial finalizer. The split factor and block sizes are not tuned for this specific small-channel, large-spatial shape.

### Why the oracle is 1.65x faster:
1. **Better parallelism**: 16 splits across 294K elements gives 128*16=2048 thread blocks, fully utilizing the GPU. Inductor's generic split factor underutilizes the available parallelism for this shape.
2. **Fused pointwise**: The erf/exp/mul chain is computed inline during the partial reduction, avoiding any intermediate materialization.
3. **Shape-specialized tuning**: The oracle's SPLIT_CHUNK and PARTIAL_BLOCK are tuned for this exact shape, while Inductor uses a one-size-fits-all heuristic.

## Fix Assessment
**Reduction scheduler enhancement** -- The reduction scheduler should select a shape-specific cooperative split-K plan for small-channel (C <= 256), large-spatial (N*H*W > 100K) reductions with fused pointwise producers. The split factor should be tuned based on the reduction size and available SMs.
