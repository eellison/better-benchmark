## Diagnosis

The repro takes one contiguous `float32[128, 768]` CUDA input, views it as
`float32[128, 1, 768]`, reduces dim 0 with `keepdim=True`, and returns the
contiguous `float32[1, 1, 768]` column sums. The oracle preserves the full
scope by allocating that exact output layout and directly reducing each input
column with one shape-specialized Triton kernel.

Gap diagnosis (classification: `BANDWIDTH_BOUND`): the oracle computes the full
view-and-sum scope as one direct Triton column reduction over `mm_9`, whereas
Inductor already emits an equivalent one-launch reduction for the same
metadata-only view and tiny 0.39 MB input; Inductor cannot expose a meaningful
scheduler-fusion, scatter-reduce, cooperative split-K, algebraic-elimination,
recompute-fusion, or new-pattern opportunity because this isolated graph has no
extra producer or consumer to fuse and the remaining work is the required input
read, f32 accumulation, output store, and launch; the fixing classification is
`BANDWIDTH_BOUND`, so only generic small-reduction launch/codegen overhead
reductions would improve both paths.
