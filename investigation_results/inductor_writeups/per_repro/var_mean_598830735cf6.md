## Root Cause

The B200 (sm_100, major=10) has a split-reduction no_split_threshold of 524288.
The BN var_mean has numel_hint=64 (channels) and reduction_numel=401408.
Since 401408 < 524288, splitting was disabled, leaving only 64 CTAs for 148 SMs
(43% GPU utilization). The threshold was calibrated for scalar reductions
(numel_hint=1) but is too conservative for multi-output reductions that can't
saturate the GPU.

## Fix

Commit `5d4ce5df93b` in `/tmp/pytorch-work` on branch `pr-184905`.

When `numel_hint < num_sm` on major >= 10 GPUs, fall back to the legacy threshold
of 8192 for the no-split decision. This enables split reductions that provide
64 * split_factor CTAs instead of just 64, properly saturating the GPU.

- Config gate: `config.split_reductions_for_undersaturated_gpu = True` (default enabled)

## Oracle Structure (3 kernels)

1. Partial BN stats (split across CHANNELS x NUM_PARTIALS = 64x197 CTAs)
2. Finalize stats + running mean/var update (64 CTAs)
3. Fused BN+ReLU+maxpool: reads original input, recomputes BN inline at each
   pool position, writes only final pooled values (no BN buffer materialized)

## Inductor After Fix (4 kernels)

1. Split partial welford reduction (64 x split_factor CTAs)
2. Final welford combine + running stat copies (64 CTAs)
3. BN+ReLU pointwise (materializes [32,64,112,112] buffer)
4. Maxpool (reads materialized buffer at 9 positions)

## Remaining Gap (1.29x)

The residual gap comes from:
- Inductor materializes the full BN+ReLU buffer (100MB write + read) while the
  oracle inlines BN into the pool (no intermediate buffer)
- The `inline_cheap_producers_into_stencils` config addresses this but is
  currently disabled because the L2-cached materialized buffer is actually faster
  to read from than recomputing 9x (the write is overlapped with reduction work)
- A proper fix would fuse the BN epilogue INTO the split reduction's final stage
  (writing BN output alongside stats), then have the pool read from L2-hot buffer
