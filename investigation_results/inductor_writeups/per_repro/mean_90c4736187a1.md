# mean_90c4736187a1 - ResNet BN Residual Spatial Mean

## Benchmark Results
- Oracle: 11.52 us
- Compiled: 12.48 us
- Ratio: 1.083x (oracle wins)

## Classification
NEW_PATTERN - BN residual ReLU spatial-mean with fp16 round boundary

## Root Cause

The oracle computes the full ResNet inference BN-affine, required fp16 round, residual fp16 add, NaN-preserving ReLU, and 7x7 spatial mean in one shape-specialized row-tiled reduction kernel. Input is f16[32, 2048, 7, 7] for the BN path plus f16[32, 2048, 7, 7] for the residual (relu_147).

The computation is:
1. BN-affine: (conv - mean) * rsqrt(var+eps) * weight + bias (in f32)
2. Convert to f16 (explicit round boundary)
3. Add residual (relu_147) in f16
4. ReLU
5. Spatial mean over 7x7 = 49 elements
6. View to [32, 2048]

Inductor fuses all these operations into one kernel but:
1. Uses sqrt+reciprocal instead of rsqrt for BN normalization
2. The total gap (1.083x) is small, suggesting the fusion is nearly correct

The 32*2048 = 65536 rows each reduce 49 elements. The per-row BN computation overhead (sqrt vs rsqrt) accounts for the small gap.

## Kernel Count
- Oracle: 1 kernel (BN-affine + fp16 + residual add + ReLU + spatial mean)
- Inductor: 1 kernel (same fusion)

## Config Exploration
- `combo_kernels = True`: no effect (already fused)
- `coordinate_descent_tuning = True`: minimal improvement expected
- The 1.083x gap is within the range explained by rsqrt canonicalization

## Why Inductor Cannot Do This Today

The gap is primarily the rsqrt vs sqrt+reciprocal issue common to all BN-inference patterns. The fusion itself is correct (Inductor successfully fuses BN + residual + ReLU + spatial mean into one kernel). The remaining delta is purely arithmetic.

Note: The oracle description mentions "does not preserve the explicit fp16 cast before the residual add" as an Inductor issue, but the 1.083x gap suggests the cast ordering is not a major performance concern here.
