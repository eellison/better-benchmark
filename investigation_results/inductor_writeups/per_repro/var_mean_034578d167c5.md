# var_mean_034578d167c5

## Compile: 18.88us, Oracle: 15.97us, Gap: 1.182x

## Diagnosis: SCHEDULER_FUSION

## Root cause: The oracle computes the Whisper residual-add LayerNorm in one row kernel that reads the strided residual input (stride=[576000, 1, 1500]) directly, performs fp16->fp32 cast, var_mean, affine, fp16 cast, and outputs [12000,384] contiguously. Inductor already fuses this into 1 kernel (`triton_red_fused_add_clone_convert_element_type_mul_rsqrt_sub_var_mean_view_0`), so the kernel count matches.

The 18% gap comes from:
1. Inductor uses Welford reduction with a loop (R0_BLOCK iteration) for hidden=384, while the oracle uses a persistent approach with BLOCK_H=512 (next power of 2 above 384) fitting in one tile
2. Inductor reads the strided input twice (once for Welford mean/variance pass, once for the normalization pass) due to the non-persistent reduction strategy
3. The oracle avoids the `clone` by reading the strided input once into registers and computing everything in a single pass

## Kernel count
- Inductor: 1 kernel (red_fused_add_clone_convert_element_type_mul_rsqrt_sub_var_mean_view)
- Oracle: 1 kernel (persistent single-pass row kernel)

## Config exploration results
- multi_kernel=0: 40.61us, multi_kernel=1: 44.90us, multi_kernel=2: 44.87us, multi_kernel=3: 44.17us
- No config helps; multi_kernel makes it worse
- coord_descent_tuning already enabled

## Fix path: This is a persistent-vs-non-persistent reduction choice issue in `choices.py`. For hidden=384 (next_pow2=512), the entire row fits in registers. The norm template should:
1. Detect that hidden dimension fits in a single RBLOCK tile
2. Emit a persistent (single-pass) reduction instead of a two-pass Welford loop
3. Avoid re-reading the strided input for the normalization epilogue

This is the same root cause as var_mean_5989a301601b (hidden=1280, BLOCK_H=2048) -- persistent row reductions for small hidden dimensions.
