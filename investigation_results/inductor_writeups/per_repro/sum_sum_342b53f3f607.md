## Scope

The diagnostic oracle covers the full compiled `repro.py` scope. It consumes the
same five original inputs, treats the clone/copy/slice as channels `80:160` of
the `[512, 160, 7, 7]` input, computes `sum(slice)` and
`sum(slice * (arg469 - arg479))` over `[0, 2, 3]`, and returns the same
contiguous `[512, 80, 7, 7]` tensor plus contiguous `[80]` vector with matching
dtype and strides.

## Gap Diagnosis

The oracle differs from Inductor by forcing a full-scope multi-output reduction
schedule: one Triton split reduction shares the sliced producer and centered
producer across both channel sums, and one Triton finalizer consumes those sums
to write the dependent BN-backward tensor epilogue and vector output. This is a
valid full-scope diagnosis, not a reduction-only subset microbenchmark.

The schedule does not establish a true floor. On this run it is faster than both
required local compile configs, but it is slower than the historical best compile
time from the queue. Inductor therefore cannot use this artifact as evidence of
a remaining scheduler-fusion floor; the computation is dominated by reading the
two full input tensors and writing the full output, and the tuned historical
generated schedule already matches or beats the hand-written full-scope Triton
paths tried here. The actionable Inductor change is no new optimization from
this repro; any future fix would need a faster full-scope schedule that beats
both required local configs and the historical best.
