# mean_f7170c220032 - Virtual Cat BN Spatial Mean (GhostNet)

## Classification: SCHEDULER_FUSION
## True Floor: Yes
## Actionable: Yes

## Benchmark Results
- Oracle: 69.34 us
- Compiled: 77.70 us
- Ratio: 1.12x

## Root Cause

The oracle computes BN-inference affine + ReLU for one branch, virtually concatenates it with an existing ReLU output (the other branch), and reduces the combined [512, 960, 7, 7] spatial mean directly to [512, 960, 1, 1] without materializing the concatenated intermediate.

Inductor produces 1 kernel that handles the entire computation. However, Inductor must materialize the cat before reducing. The pattern is:
```
[relu_37: f32[512, 480, 7, 7], BN+ReLU(conv): f32[512, 480, 7, 7]] -> cat(dim=1) -> [512, 960, 7, 7] -> mean(dim=[2,3])
```

The oracle uses 2 kernels:
1. A kernel that computes BN-affine + ReLU + spatial mean for the new branch, writing to channels [480:960] of the output
2. A kernel that computes spatial mean of the already-computed relu_37 branch, writing to channels [0:480] of the output

This avoids materializing the [512, 960, 7, 7] = 512*960*49*4 = ~96 MB concatenated tensor.

## Kernel Count
- Inductor: 1 kernel (fused pointwise + cat + reduction)
- Oracle: 2 kernels (one per branch, each does spatial mean independently)

## Config Exploration
- `coordinate_descent_tuning=True`: Already enabled
- `combo_kernels=True`: Does not help - the cat materializes before reduction regardless
- `multi_kernel=3`: 79.87 us - slightly worse than baseline

## Why Inductor Cannot Do This Today

Inductor's scheduler treats `aten.cat` as a node that must produce a materialized output before any consumer can read it. When the consumer is a spatial reduction (mean over H,W), this forces writing and re-reading the full concatenated tensor.

The oracle recognizes that mean(cat([A, B], dim=C), dim=[H,W]) can be decomposed into:
- output[:, :480] = mean(A, dim=[H,W])
- output[:, 480:] = mean(B, dim=[H,W])

This is algebraically valid because mean over non-cat dimensions distributes over the cat.

## Required Enhancement (Design Doc)

**Cat-Through-Reduction Decomposition**: An FX pass or scheduler optimization that recognizes:
```
cat([t1, t2, ...], dim=D) -> reduction(dims=other_dims)
```
where the reduction is over dimensions OTHER than the cat dimension, and decomposes it into independent per-branch reductions writing to slices of the output.

This is related to the mean_d0fc206717a8 case but simpler (only 2 branches, one is already computed). It could be implemented as:
1. An FX pass in `torch/_inductor/fx_passes/post_grad.py` that splits `mean(cat(...))` into per-branch means
2. Or a scheduler enhancement that virtualizes the cat when the sole consumer reduces over orthogonal dimensions
