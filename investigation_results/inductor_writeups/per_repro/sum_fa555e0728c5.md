## Full-Scope Contract

The artifact consumes the same `mm_708`, `arg1313_1` boolean mask, scalar
`full_1`, and three shape parameters as `repro.py`. It returns the same two
outputs: a `float32[512, 32768]` transpose view with stride `(1, 512)` and a
contiguous `float32[512]` column sum. The timed path is Triton-only for the
oracle computation: one tiled `where` materialization kernel also writes row
block partial sums, and one small finalizer reduces those partials.

## Gap Diagnosis

The oracle differs from Inductor by explicitly sharing the `where(mask, full,
mm)` producer between the returned transposed backing storage and the `[512]`
sum side output, so the full tensor write and reduction partial accumulation
happen in one streamed Triton pass. Inductor cannot expose a meaningful
remaining optimization for this captured shape today because the best compiled
variants already run as two bandwidth-dominated kernels with the same practical
traffic envelope: read the 16M-element `float32` input, read the 16M-element
bool mask, write the 16M-element returned tensor, and finalize tiny column
partials. The fix class is `BANDWIDTH_BOUND`; no Inductor scheduler/codegen
change is justified unless a future full-scope oracle beats both local required
configs and the `33.76000002026558 us` historical best.
