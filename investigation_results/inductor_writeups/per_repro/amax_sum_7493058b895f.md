## Root Cause

The oracle computes T5 attention softmax by proving that the `iota(2048) >= 0` mask is tautologically true (all values in arange(2048) are >= 0), eliminating the where/full/add zero-bias path entirely, and running a single row-softmax kernel over [8, 2048, 2048] fp16 scores.

Inductor already fuses this into a single online softmax kernel (`triton_red_fused_add_convert_element_type_div_exp_expand_full_ge_prepare_softmax_online_sub_view_where_0`) but retains the redundant iota/ge/where/add logic inside the kernel body. The 1.058x gap comes from slight autotuning differences in block size selection.

- **Inductor**: 1 kernel (online softmax, dynamic RBLOCK with coordinate descent tuning)

## Analysis

The gap is marginal (5.8%) and disappears entirely with `triton.multi_kernel = 3`. Inductor already fuses everything into one kernel and uses online softmax. The algebraic elimination (removing the tautological mask) would reduce instruction count slightly but does not cause a meaningful perf gap here.

## Design Doc

**Ideal fix**: Shape-aware predicate simplification in `torch/_inductor/fx_passes/` that proves `arange(N) >= 0` is always true and eliminates the zero-bias addition before scheduling. This would clean up generated code but given the gap is < 6% and disappears with autotuning, priority is low.
