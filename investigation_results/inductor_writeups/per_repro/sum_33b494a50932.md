## Diagnosis

The oracle computes the full exact-GELU derivative producer once, writes the required base buffer for the returned transposed view, and accumulates column partials in the same Triton program before a finalization reduction. Inductor emits a separate pointwise materialization followed by two reduction kernels that re-read the materialized producer.

The repro computes over [8192, 3072] f32:
1. GELU derivative: `mul(mm, 0.5*(1+erf(arg*0.707)) + arg*0.399*exp(-0.5*arg^2))`
2. Return transposed permute [3072, 8192] (backing store)
3. Return sum(result, dim=0) -> [3072]

## Root cause

Inductor cannot fuse a transposed materialization (store in column-major order) with a sibling column reduction (sum over rows) because the multi-output kernel pattern for store+reduce is not supported. The oracle iterates once over the [8192, 3072] data, computes the GELU derivative, stores to the backing buffer, and accumulates the column partial sum -- achieving one read per element.

Inductor schedules:
- Kernel 1: pointwise GELU derivative -> write [8192, 3072]
- Kernel 2: transpose copy for [3072, 8192] return
- Kernel 3: column reduction sum(dim=0) reading the same buffer again
