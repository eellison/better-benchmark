# sum_a630880e528a — SCHEDULER_FUSION

## Summary
- **Model**: torchbench_LearningToPaint_train_001
- **Pattern**: Pointwise `arg126 * arg104 * (1 - arg104)` producing both a permuted `[65, 1024]` output and a sibling column sum `[65]`
- **Oracle**: `oracle_sum_permute.py`
- **Ratio**: 1.345x (oracle 7.14us vs compile 9.60us)
- **Classification**: SCHEDULER_FUSION

## Root Cause

The oracle computes both outputs (permuted materialization + column sum) in a single kernel launch with 65 blocks (one per column). Each block loads all 1024 rows for its column, computes `b * a * (1 - a)`, writes the result to the permuted output buffer, and accumulates the column sum — all in one pass with BLOCK_M=1024.

Inductor schedules 3 separate kernels:
1. `triton_poi_fused_mul_sub_0`: Pointwise kernel computing `arg126 * arg104 * (1 - arg104)` and materializing to a `[1024, 65]` buffer (66,560 elements)
2. `triton_red_fused_sum_1`: Reduction kernel reading the materialized buffer and computing partial sums (520 x-blocks, 128 reduction elements each)
3. `triton_per_fused_sum_2`: Finalize the column sums from 8 partial tiles

The permuted output `[65, 1024]` is a view of the `[1024, 65]` buffer (just different strides), so Inductor must write it row-major then the consumer reads it column-major. The sum reduction also reads from the same intermediate buffer.

## Kernel Count
- **Oracle**: 1 kernel
- **Inductor**: 3 kernels

## Analysis

The core issue is that Inductor's scheduler does not fuse a layout-changing (permuted) output with a compatible sibling reduction. Specifically:
- The pointwise expression `b * a * (1 - a)` has two consumers: a permute (which is just a stride change) and a sum over dim 0
- Because the permute materializes the buffer in row-major layout, the scheduler treats it as a realized node that blocks fusion with the reduction
- The oracle shows that tiling by column (65 blocks, each processing all 1024 rows) naturally provides both outputs: the column-major write for the permute and the reduction for the sum

The 34.5% gap is significant for such a small problem size (66K elements). The overhead is 3 kernel launches + redundant memory traffic (write intermediate, read it twice for sum).

## Design Doc

**Why Inductor cannot fix this today**: The scheduler does not recognize that when a pointwise node has two consumers — one that just changes strides (permute) and one that reduces along the same dimension — both can be served by a single kernel tiled along the non-reduced axis. The permute is a "free" operation (just a stride reinterpretation), but the scheduler treats it as requiring materialization, which forces the intermediate to be written, breaking fusion with the reduction.

**Needed enhancement**: Teach the scheduler that `permute` consumers that share the same producer as a reduction can be co-scheduled: tile the kernel along the permuted output's leading dimension, and within each tile, both write the materialized result (for the permute) and accumulate the reduction. This is a multi-output fusion pattern where the "layout change" output and the "reduction" output share the same data source and can be computed in one pass.

**Affected repro count**: This pattern (pointwise -> permute + sum) appears in multiple LearningToPaint and similar repros (sum_7953a8b0bbba, sum_acbec27412d2 share the oracle_sum_permute classification).
