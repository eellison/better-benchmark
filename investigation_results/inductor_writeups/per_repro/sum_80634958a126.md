## Full-Scope Contract

The oracle consumes the same four float32 tensors and five shape parameters as
`repro.py`:

```text
mm_142:  f32[4096, 4096]
mul_341: f32[8, 512, 4096]
mm_144:  f32[4096, 4096]
mm_146:  f32[4096, 4096]
```

It computes the complete repro expression:

```text
producer[row, col] = ((mul_341.view(4096, 4096)[row, col] + mm_142[row, col])
                      + mm_144[row, col]) + mm_146[row, col]

out0 = producer.t()          # shape [4096, 4096], stride (1, 4096)
out1[col] = sum_row producer[row, col]
```

The timed Triton path is not a reduction subset. It writes the materialized
producer backing storage for the returned transpose view and writes per-column
partial sums from the same tile pass, then finalizes the vector in a second
Triton kernel.

## Gap Diagnosis

The oracle differs from Inductor by making the producer sharing explicit: it
computes the three-add value once, stores it to the contiguous backing tensor
whose transpose is returned, and accumulates the column-sum partials from the
same loaded tile. Inductor's generic scheduling does not form this exact
materialized-view plus reduction template today; it handles the returned
transpose backing storage and the reduction through its existing generic
lowerings.

The relevant Inductor change would be a scheduler/codegen template that shares a
pointwise producer between a materialized non-contiguous view return and a
reduction. The historical-best gate shows that this repro does not currently
provide a usable true-floor oracle for that change; treat it as
`BANDWIDTH_BOUND`/already-at-floor for queue integration.
