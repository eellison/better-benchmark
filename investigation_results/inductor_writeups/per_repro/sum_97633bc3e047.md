## Root Cause

The oracle splits the 128x96x96 reduction domain (1,179,648 elements per channel) into 32 cooperative split-K tiles of 4 batches each (4x96x96 = 36,864 elements per split). Each partial reduction kernel computes the full GELU-gradient pointwise expression (erf, exp, multiply) and accumulates a per-channel partial sum. A tiny 16-element finalizer sums the 32 partials.

Inductor schedules this as a two-kernel generic reduction:
1. `triton_red_fused_add_erf_exp_mul_sum_0`: 576 x-blocks (16 channels x 36 splits), each reducing 32768 elements with BLOCK_R loop
2. `triton_per_fused_add_erf_exp_mul_sum_1`: 16-element finalizer summing 36 partials

## Kernel Count
- **Oracle**: 2 kernels (partial reduce + finalize)
- **Inductor**: 2 kernels (same structure)

## Analysis

Both use two kernels with a similar split-K pattern. The performance gap (10.3%) comes from:
1. The oracle uses a fixed BLOCK_R=1024 with 8 warps and 3 pipeline stages, optimized for this specific reduction size
2. Inductor's generic reduction scheduler picks split factor 36 (from 128*96*96/16 = 1,179,648 elements split into 32768 per block) which yields 576 blocks vs the oracle's 512 (16x32)
3. The oracle's split factor (32) is tuned to give exactly 36,864 elements per split which fits well in registers; Inductor's heuristic picks a different split that is slightly less efficient

The aggressive split threshold (8586e404cc8) appears to have caused a regression for this
16-channel GELU-gradient pattern. The compiled time went from 52.99us to 61.31us while the
oracle time improved slightly (48.06us -> 46.05us). The new split factor may be suboptimal
for this small-output-channel shape with very large reduction domain (1.18M elements/channel).

## Design Doc

The gap has grown from 10.3% to 33.1% after the aggressive split threshold commit. The oracle's
hand-tuned split factor of 32 (yielding 36,864 elements/split with BATCHES_PER_SPLIT=4) is now
significantly more efficient than Inductor's auto-selected factor. The fix would be to improve
the cooperative reduction split-factor search in `choices.py` to consider shapes where the output
channel count is very small (16) relative to the total reduction domain (1.18M), favoring
power-of-2 split factors that align with batch boundaries. This regression makes this fix higher
priority than before.
