## Root Cause

The oracle materializes `tangents_1 / 2` once for the returned `[1000, 128]` permute view and reduces those same scaled values once to produce both duplicate `[1000]` sum outputs. The two `sum(dim=0)` operations are identical (same input, same reduction dims) but Inductor schedules them as separate reductions.

Shape: [128, 1000] with reduction over dim=0 (128 elements). The two sums are mathematically identical.

- **Inductor**: Likely 2-3 ops (div + 2 separate reductions + permute)

## Design Doc

This is a simple CSE (common subexpression elimination) problem. The two `sum.dim_IntList(div_tensor, [0], True)` calls are identical but produce separate output nodes because they serve different return positions. Inductor's graph-level CSE should detect and merge these, allowing both returned sum outputs to alias a single computed buffer.

Fix: Enhance the post-grad CSE pass to detect identical reduction nodes (same input, same dims, same keepdim) and merge them, with the output contract maintaining both references to the shared result. This is trivial in principle but needs care around output aliasing semantics.

Given the 0.6us absolute gap and 1.10x ratio, this is very low priority.
