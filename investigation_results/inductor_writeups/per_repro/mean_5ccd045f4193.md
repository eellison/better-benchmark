# mean_5ccd045f4193 - Sequence Mean (Whisper)

## Benchmark Results
- Oracle: 9.28 us
- Compiled: 12.96 us
- Ratio: 1.397x (oracle wins)

## Classification
SCHEDULER_FUSION - view + middle-dimension mean not specialized

## Root Cause

The oracle computes `view(f16[12000, 256] -> [8, 1500, 256])` followed by `mean(dim=1)` in a single Triton reduction kernel that tiles by (batch, hidden_block), with each program reducing 1500 sequence elements for a block of 8 hidden features at a time. It uses a fixed BLOCK_SEQ=2048 with masking and stores only the final f16 [8, 256] output.

Inductor lowers this into a two-kernel split reduction:
1. `triton_red_fused_mean_view_0`: xnumel=24576, r0_numel=125 - this splits the 1500 elements into partial reductions across 12 chunks (1500/125=12), producing 24576 partial sums (2048*12)
2. `triton_per_fused_mean_view_1`: xnumel=2048, r0_numel=12 - reduces the 12 partial sums per output element

The oracle uses 1 kernel with a large BLOCK_SEQ=2048 tile that covers all 1500 elements in one pass. Inductor splits this into a two-pass reduction because its heuristics determine that reducing 1500 elements in a single tile is too large for the persistent reduction threshold at the given xnumel.

The key issue is that Inductor's reduction scheduler doesn't specialize for this case where:
- The reduction dimension (1500) fits in a single block (2048)
- The output is small (8*256 = 2048 elements)
- A single-pass persistent reduction would be optimal

## Kernel Count
- Oracle: 1 kernel (fused view + mean over seq dim)
- Inductor: 2 kernels (split reduction)

## Config Exploration
- `combo_kernels = True`: no effect (not a fusion issue)
- `coordinate_descent_tuning = True`: enabled by default, doesn't fix split reduction decision
- The split reduction choice is driven by `choices.py` heuristics

## Why Inductor Cannot Do This Today

Inductor's reduction scheduler (in `choices.py`) decides to split this reduction because it applies general heuristics that don't account for the fact that:
1. The view is metadata-only (no actual data movement)
2. The reduction dim (1500) fits in a single power-of-2 block (2048)
3. The total output is tiny (2048 elements), making kernel launch overhead of the second kernel relatively expensive

The scheduler sees xnumel=2048 with reduction=1500 and decides a split factor is better than a persistent reduction, but for this shape the two-kernel overhead plus the intermediate buffer write/read dominates.

## Design Doc

**Fix location**: `/tmp/pytorch-work/torch/_inductor/choices.py` (should_use_cooperative_reduction / split factor selection)

**Enhancement needed**: When the reduction dimension fits within a single BLOCK_R (power-of-2 up to some limit like 2048) and the output numel is small enough to saturate SMs without splitting, prefer a single-pass persistent reduction. The heuristic should account for:
- Whether a single block can cover the full reduction (rdim <= max_block_r)
- The ratio of kernel launch overhead vs. the intermediate buffer traffic

**Affected repro count**: This pattern (view + middle-dimension mean with moderate reduction size 1000-2000) appears in transformer/attention models like Whisper. Likely 2-5 similar repros in the corpus.
