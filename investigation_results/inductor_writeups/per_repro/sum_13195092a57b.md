# sum_13195092a57b

## Compile: 16.26us, Oracle: 13.28us, Gap: 1.224x

## Classification: PERMUTE_SIDE_SIBLING

## Root Cause

The oracle streams through the contiguous [8, 1500, 384] (= [12000, 384]) input once, writing both the [384, 12000] transposed clone output AND accumulating the [384] column sum atomically in a single kernel pass.

Inductor generates 3 kernels:
1. `triton_poi_fused_clone_0`: materializes the clone (copy to new buffer)
2. `triton_red_fused_clone_sum_view_1`: partial column reduction
3. `triton_red_fused_clone_sum_view_2`: final column reduction

The fundamental issue is that Inductor cannot fuse a layout-changing materialization (the permute [384, 12000] side output, which requires writing in transposed order) with a sibling reduction (sum over dim=0 of the same [12000, 384] data). The scheduler treats these as separate consumers of the same producer.

## Kernel Count
- Oracle: 1 kernel (copy + atomic column sum)
- Inductor: 3 kernels (clone, partial reduce, final reduce)

## Fix Assessment: Design doc

The scheduler needs a "multi-output materializing reduction" template: when a producer is consumed by both a layout-changing store (permute/transpose) AND a reduction, emit a single kernel that iterates over the producer data, writing the transposed output AND accumulating column partials with atomic_add.

### Why it's hard:
The permute output requires data stored in [384, 12000] layout (stride=(1, 384)), while the reduction sums along dim=0 of [12000, 384]. The oracle solves this by tiling [BLOCK_M=256, BLOCK_N=32] over the natural [12000, 384] layout, storing each tile in the output buffer at the same natural offsets (since the permuted view's physical storage is the same as the unpermuted buffer), and accumulating column partials atomically.
