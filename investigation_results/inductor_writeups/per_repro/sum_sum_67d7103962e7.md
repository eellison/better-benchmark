## Root Cause

The oracle computes a MobileNet-style batch-norm backward by fusing the two sibling channel reductions (`sum(add, [0,2,3])` and `sum(add * centered, [0,2,3])`) into a cooperative split-K approach (C*N = 24*128 = 3072 blocks for partial reduction), then finalizes both channel stats, and writes the full `[128, 24, 56, 56]` gradient tensor using those finalized scalars.

Inductor generates a single persistent reduction kernel with only 24 x-elements (one per channel) and a massive reduction dimension of 401,408 elements (N*H*W = 128*56*56). This means only 24 thread blocks active on the GPU, severely underutilizing parallelism. The kernel also reads the input data twice (once for reduction, once for epilogue).

## Kernel Count
- **Oracle**: 3 kernels (partial_sum_sum: C*N grid, finalize_stats: C grid, write_full_output: C*tiles grid)
- **Inductor**: 1 kernel (`triton_red_fused_add_mul_sub_sum_unsqueeze_0` with only 24 blocks)

## Config Exploration
Inductor already fuses everything into a single kernel with `combo_kernels=True` and `coordinate_descent_tuning=True`, but the single-kernel approach with 24 blocks is not GPU-efficient for this shape. The problem is the reduction strategy, not fusion.

## Design Doc

The core issue is that Inductor's reduction codegen chooses a single-pass persistent reduction with XBLOCK=24 for the channel reduction. With only 24 thread blocks, GPU occupancy is terrible on modern hardware (B200 has 192 SMs).

The oracle's split-K approach (3072 blocks for partial reduction) achieves 100x more parallelism. Inductor needs:
1. A cooperative split-K reduction strategy that splits the N dimension across multiple blocks per channel
2. A multi-output reduction template that shares the producer across sibling reductions
3. The ability to sink finalized reduction results into a dependent full-output epilogue

This is the canonical "BN-backward multi-output reduction" pattern affecting many repros. The fix belongs in `choices.py` (to select cooperative reduction) and `scheduler.py`/`codegen/triton.py` (to handle multi-output with shared producer).
