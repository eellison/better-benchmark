# mean_8b70c08405c1 - Residual RMSNorm Aliases (LLaVA)

## Benchmark Results
- Oracle: 6.94 us
- Compiled: 7.55 us
- Ratio: 1.088x (oracle wins)

## Classification
RECOMPUTE_FUSION - RMSNorm row reduction with recomputed residual-add for epilogue

## Root Cause

The oracle computes the complete residual-add RMSNorm alias scope in one Triton row kernel:
1. View mm_220[512, 4096] -> [1, 512, 4096] (metadata-only)
2. Add with add_219[1, 512, 4096] (residual connection)
3. Convert to f32
4. pow(x, 2) then mean(dim=-1) over 4096 elements
5. rsqrt(mean + 1e-6)
6. Multiply by rsqrt (normalize)
7. Convert back to f16
8. Multiply by weight arg287_1[4096]
9. Output as two aliased views [512, 4096] (same buffer, two shape params)

Inductor generates a fused reduction but uses a two-pass approach:
- First pass: accumulates mean(square) over the hidden dimension
- Second pass (or reload): reloads the residual-add result to apply normalization in the epilogue

The oracle retains the full hidden tile (4096 values) in registers across the reduction, avoiding the second pass. This is the classic RMSNorm recompute pattern.

## Kernel Count
- Oracle: 1 kernel (single-pass RMSNorm with retained tile)
- Inductor: 1 kernel (but with internal recompute/reload of activation for epilogue)

## Config Exploration
- `combo_kernels = True`: no effect
- `coordinate_descent_tuning = True`: helps tune block sizes
- The 1.088x gap is small, suggesting Inductor's recompute is only marginally worse

## Why Inductor Cannot Do This Today

Inductor's reduction scheduler does not keep the full hidden tile's producer values live across the row reduction for the normalization epilogue. For a hidden dim of 4096 with f32, this would require 16KB of register/shared memory per row - which is feasible on modern GPUs but exceeds Inductor's default register pressure limits.

Instead, Inductor recomputes the residual-add in the epilogue (loading mm_220 and add_219 again) or uses a single persistent kernel that loads twice. The small gap (1.088x) suggests Inductor's approach is nearly optimal - the second load from L2 cache is fast since the data was just read.

## Design Doc

**Fix location**: `/tmp/pytorch-work/torch/_inductor/choices.py` (persistent reduction with retained tile)

**Enhancement needed**: For RMSNorm patterns where:
- The reduction dim is the last dim (row-local)
- The normalized values are needed in the epilogue
- The tile fits in registers (hidden_dim * sizeof(dtype) < register budget)

Add a guarded "retained-tile" reduction schedule that keeps the producer values live across the reduction accumulation.

**Affected repro count**: RMSNorm appears in all LLM models (LLaMA, LLaVA, Mistral, etc.). Likely 5-10 repros in the corpus with this pattern. The small gap (1.088x) means this is low priority compared to BN-inference rsqrt fixes.
