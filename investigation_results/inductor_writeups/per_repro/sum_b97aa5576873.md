## Root Cause

The repro (from timm swin_base_patch4_window7_224 training) computes the Swin qkv layout clone and sibling column sum:
- Inputs: 3x [32768, 49, 32] / [32768, 32, 49] f32 tensors (attention QKV)
- Outputs: [401408, 384] f32 (layout clone) + [384] f32 (column sum)
- The layout involves cat of 3 permuted/viewed tensors into [3, 8192, 4, 49, 32] then flattened

### Oracle approach:
The oracle materializes the required transpose backing buffer while also emitting per-column partial sums from the same loaded tiles, avoiding a reread of the full layout output for the reduction.

### Inductor approach:
Inductor writes the clone buffer first, then rereads it through reduction kernels for the sum. Since the layout is also a returned output, the scheduler treats it as a materialization barrier.

### Why the gap is small (5.4%):
The layout materialization dominates runtime (~300 us for writing 401408*384*4 bytes = ~617 MB). The sum reduction over 401408 rows is a relatively small fraction. The oracle saves one reread of the layout for the reduction but the layout write itself is the bottleneck.
