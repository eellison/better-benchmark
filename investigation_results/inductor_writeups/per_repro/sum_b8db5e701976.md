## Root Cause

The repro (from hf_LayoutLM train) does:
1. Takes [32, 12, 512, 64] attention head output (strided: stride=(393216, 64, 768, 1))
2. permute [0,2,1,3] -> clone (contiguous) -> view [32, 512, 768] -> view [16384, 768]
3. Returns permute [768, 16384] as output 0 (transposed layout)
4. Returns sum(dim=0) [768] as output 1

This is structurally identical to sum_7953a8b0bbba: a layout-changing materialization + sibling reduction.

The oracle uses 2 kernels:
1. A fused kernel that reads the strided input, writes the [768, 16384] transposed output, AND accumulates partial sums
2. A final reduction kernel to sum the partials into [768]

Inductor uses 3 kernels:
1. `triton_poi_fused_div_expand_unsqueeze_0` - pointwise clone/permute materialization
2. `triton_red_fused_*_1` - partial reduction
3. `triton_red_fused_*_2` - final reduction

The gap is that Inductor doesn't fuse the layout copy (clone+permute) with the reduction. It materializes the clone separately, then reduces from the materialized buffer. The oracle streams once through the strided input and writes both the transposed output and partial sums.

## Fix Assessment
**Design doc** - Same fundamental issue as sum_7953a8b0bbba.

### What's needed:
A "multi-output layout-copy-plus-reduction" schedule in the scheduler that recognizes when:
1. A producer feeds both a layout-changing store and a reduction
2. The reduction's iteration space is compatible with the producer's natural iteration order
3. Writing both outputs from a single traversal saves a full buffer round-trip

### Memory savings calculation:
- Without fusion: read [32,12,512,64] strided -> write [16384,768] contiguous -> read [16384,768] for reduction
- With fusion: read [32,12,512,64] strided -> write [768,16384] transposed + accumulate partials
- Saved: one full read of [16384,768] = 48 MB at fp32

### Relevant files:
- `/tmp/pytorch-work/torch/_inductor/scheduler.py` - needs new multi-output pattern
- `/tmp/pytorch-work/torch/_inductor/ir.py` - realize_hint forces materialization before reduction can start

### Affected repro count:
At least 2 repros in this sample (sum_7953a8b0bbba, sum_b8db5e701976). The "layout copy + sibling reduction" pattern is common in attention head reshape paths.
