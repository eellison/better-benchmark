# var_mean_c408b9eb8855

- **Gap**: 1.47x (oracle 7.55us, compile 11.07us)
- **Classification**: NEW_PATTERN
- **Label**: torchbench_hf_GPT2_infer_000
- **Pattern**: GPT-2 embedding + layernorm fused with constant adjacent-position mask elimination

## What the "constant adjacent-position mask" is

The repro computes a boolean mask via this subgraph:

```
iota(1024, start=0, step=1)    -> [0, 1, 2, ..., 1023]
add(_, 0)                      -> [0, 1, 2, ..., 1023]  (no-op)
unsqueeze(_, 0)                -> [[0, 1, 2, ..., 1023]]
slice(_, 1, 0, 1)              -> [[0]]
sub(_, 1)                      -> [[-1]]
cat([[-1], [0..1023]], dim=-1) -> [[-1, 0, 1, 2, ..., 1023]]
slice(_, -1, 1, 1025)          -> [[0, 1, 2, ..., 1023]]
slice(_, -1, 0, 1024)          -> [[-1, 0, 1, ..., 1022]]
sub(a, b)                      -> [[1, 1, 1, ..., 1]]
ne(_, 1)                       -> [[False, False, ..., False]]
```

The mask checks whether adjacent position IDs differ by something other than 1.
Since positions are generated by `iota` (consecutive integers starting at 0),
adjacent differences are always exactly 1, so `ne(1)` is **always all-False**.
This is a constant output that does not depend on any model inputs.

## Inductor's generated code (3 kernels)

1. **triton_poi_fused_0** (combo kernel): Writes the scalar `-1` to `buf4` and
   the iota sequence `[0..1023]` to `buf5`. These form `buf6 = [[-1, 0, 1, ..., 1023]]`
   used later by the mask kernel.

2. **triton_per_fused_1** (persistent reduction, 1024 rows x 768 cols): The main
   embedding + layernorm kernel. Loads token IDs, gathers from word embedding table,
   loads position embeddings, adds (in fp16), promotes to fp32, computes var_mean,
   applies affine transform, casts back to fp16. This kernel is structurally similar
   to the oracle's single kernel.

3. **triton_poi_fused_ne_slice_sub_2** (pointwise, 1024 elements): Reads from `buf6`,
   computes `buf6[1+x] - buf6[x] != 1`, writes bool output. Result is always False.

## Root cause of the 1.47x gap

Two contributing factors:

1. **Dead mask computation (kernels 0 + 2)**: The mask output is algebraically constant
   (always all-False). Inductor emits two extra kernel launches to compute it:
   - Kernel 0 materializes the position-index-with-prefix array
   - Kernel 2 computes adjacent differences and ne(1)
   
   These add ~2-3us of kernel launch overhead + memory traffic for the intermediate
   `buf6` (1025 int64 values = 8.2KB written then read).

2. **Generic layernorm codegen in kernel 1**: Compared to the oracle's dedicated
   row kernel, Inductor's kernel includes extra `device_assert` for index bounds,
   redundant `broadcast_to` ops, and two separate sum reductions (one for mean, one
   for variance) rather than a single-pass Welford-style approach. Based on the
   related repro var_mean_9ded65850d80 (pure embedding+layernorm, no mask), this
   accounts for roughly 1.11x overhead by itself.

   Estimated breakdown:
   - Layernorm overhead: 7.55us * 1.11 = ~8.4us (kernel 1 alone)
   - Mask overhead: 11.07 - 8.4 = ~2.7us (kernels 0 + 2 + launch costs)

## Why constant folding does not eliminate this today

Inductor's `ConstantFolder` (in `torch._inductor.constant_folding`) is called with
`skip_constructors=True`. The comment says: "skip constructors, since inductor
generates optimal code for them already and turning into tensor would result in an
additional global memory read."

The `prims.iota` node has no tensor inputs (only scalar constants), so it is
classified as a "constructor" and skipped (line 149-157 of ConstantFolder). Since
iota is skipped, all downstream operations that depend only on it (add, unsqueeze,
slice, sub, cat, slice, slice, sub, ne) also cannot be folded because the folder
never propagates a constant value for the iota node.

## Whether an FX pass could handle it

**Yes.** Two viable approaches:

### Approach A: Relax skip_constructors for small constant subgraphs

If a constructor's entire downstream subgraph (up to a graph output) consists only
of other constructors/constants and produces a small output (e.g., 1024 booleans =
1KB), it should be constant-folded rather than skipped. The "additional global memory
read" concern is negligible for 1KB outputs versus the 2-3 kernel launches saved.

### Approach B: Pattern-match iota-derived constant mask elimination

A pre_grad or post_grad FX pattern that matches:
```
iota -> [add(0)] -> unsqueeze -> slice(0:1) -> sub(1) -> cat(_, iota) 
  -> slice(1:N+1) & slice(0:N) -> sub -> ne(1)
```
and replaces the ne output with `torch.zeros(..., dtype=bool)`.

This is a GPT-2-specific pattern (and GPT-Neo, which uses the same position
embedding scheme) but it hits at least 8 repros in the corpus.

### Approach C: Extend dead code elimination

If the compiler can prove a subgraph's output is a compile-time constant (by
attempting evaluation on shape-only inputs), replace it with a constant buffer.
This is a generalization of Approach A.

## How many repros might benefit

- **1 repro with this exact oracle**: var_mean_c408b9eb8855 (1.47x gap confirmed)
- **7 additional repros** with `iota + ne_scalar` pattern but no oracle yet:
  var_mean_1b02c576e1eb, var_mean_6d9f14e83fbb, var_mean_a88cd01397d1,
  var_mean_06924cc70cb4, var_mean_2f7685c1567b, var_mean_bd299b8b72c8,
  var_mean_d3e9f404842a
- **8 embedding+layernorm oracle repros** (var_mean_0990c69ae9bb, _1fad4754680a,
  _9ded65850d80, _a4b0facb2977, _c3179bad0f03, _c4eed3e5da72, _d10ed8104f03,
  _ef88cc4744f1) that share the same embedding+layernorm pattern family and would
  benefit from a dedicated lowering template even without the mask.

**Total: 8 confirmed repros with the mask pattern, 16+ in the broader embedding+layernorm family.**

The mask elimination alone (Approach A or B) would save ~2-3us per invocation for
the 8 mask-bearing repros. Combined with a dedicated embedding-layernorm template
(the broader NEW_PATTERN fix), the full family of 16+ repros would benefit.

## Relation to embedding+layernorm family

This is a superset of the `oracle_embedding_layernorm.py` pattern seen in 8 other
repros (some already at floor with ~1.11x gap). The `_mask` variant adds the
constant-output boolean computation that inflates the gap from ~1.11x to ~1.47x.
The var_mean_9ded65850d80 writeup documents the pure embedding+layernorm case
where Inductor already emits a single kernel but with suboptimal generic codegen.
