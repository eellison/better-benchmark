# sum_5a4992885bd6 — SCATTER_REDUCE

## Summary
- **Model**: torchbench_hf_Longformer_train
- **Pattern**: Longformer sliding-window backward with structured scatter, dropout mask, edge masks, and skewed output
- **Ratio**: 1.063x

## Operation
Takes `[96, 256, 768]` f32 BMM output plus multiple bool/f32 masks and produces `[72, 512, 512]` f32 output through an extremely complex chain:
1. Views/squeezes the BMM output into `[24, 4, 256, 768]` chunked form
2. Applies structured slice_scatter to skew the chunks (Longformer sliding window pattern)
3. Pads and reshapes into `[2, 12, 1024, 513]` attention layout
4. Applies dropout mask, attention mask, and triangular edge masks
5. Computes `sum(dim=-1)` for softmax backward normalization (fma pattern)
6. Applies more slice_scatter operations for edge handling
7. Final de-skew into `[72, 512, 512]` output

## Kernel Count
- Inductor: 5 kernels
- Oracle: 2 kernels (row-sum + gather-mask-scatter-elimination)

## Benchmark
- Oracle: 187.84 us
- Compiled: 199.74 us
- Ratio: 1.063x

## Root Cause Analysis

Inductor generates 5 kernels because the complex slice_scatter/select_scatter tree creates
multiple materialization points that break fusion:

1. **triton_red_0**: Main reduction + pointwise work (fma softmax backward)
2. **triton_poi_1**: Layout conversion / scatter for intermediate
3. **triton_poi_2**: Small helper for mask/edge computation  
4. **triton_poi_3**: Slice_scatter assembly for edge handling
5. **triton_poi_4**: Final constant_pad_nd + view output

The oracle reduces this to 2 kernels by:
- Computing row sums from source rows directly (avoiding intermediate scatter)
- Mapping source chunks to final output positions algebraically (no materialized scatter)

**Key inefficiency**: The slice_scatter chain creates multiple `full(0) -> slice_scatter -> slice_scatter` patterns that Inductor materializes as separate buffers, each requiring a kernel launch and memory traffic.

## Why Inductor Cannot Do This Today

1. **Nested slice_scatter is opaque**: The scheduler does not recognize that `slice_scatter(slice_scatter(full(0), A, dim, lo, hi), B, dim, lo2, hi2)` can be computed as a direct conditional store without materializing intermediates.

2. **Skew pattern not recognized**: The Longformer "skew" operation (reshape -> pad -> slice to create a diagonal-shifted view) is a structured index mapping that Inductor treats as generic scatter.

3. **Edge-handling slice_scatter breaks fusion**: The pattern of zeroing out edge blocks via `copy(slice, full(0))` -> `slice_scatter` forces materialization of intermediate tensors.

## Fix Direction

This is a highly model-specific pattern (Longformer sliding-window attention backward). The general fix would be:
- File: `/tmp/pytorch-work/torch/_inductor/fx_passes/post_grad.py`
- Recognize chained `slice_scatter` patterns feeding into reductions and algebraically simplify them

However, given the small gap (6.3%) and extreme complexity of the pattern, this may not be worth a dedicated fix. The Longformer architecture is relatively niche.

## Conclusion
Small gap on a very complex pattern. The 5-kernel vs 2-kernel difference is mostly due to
Longformer-specific scatter patterns that are hard to generalize. The absolute time difference
is only ~12 us.
