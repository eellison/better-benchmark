# sum_sum_e9338369070e

## Classification

COOPERATIVE_SPLIT_K

## Model/Pattern

- Model: `timm_ghostnet_100` (training, batch normalization backward)
- Pattern: Two sibling `sum([0, 2, 3])` reductions over a sliced `[512, 40, 14, 14]` view of an `add` result, followed by a BN-backward-style affine epilogue producing a full `[512, 40, 14, 14]` output (channels-last) and a `[40]` side vector.
- Reduction dimension: 512 * 14 * 14 = 100,352 elements per channel
- Output x-dimension: 40 channels

## Gap

- Oracle: 43.97 us
- Default compile: ~130 us (2.96x gap)
- Best config (`force_cooperative_reductions=True`): ~72 us (1.64x gap)
- Remaining gap after best config: 1.64x

## Inductor Behavior (Default)

Inductor generates a single `triton_red_fused_*` kernel with:
- Grid: `(40,)` -- one CTA per channel
- Two sequential for-loops over the reduction dimension (100,352 elements):
  - Pass 1: computes `sum_x` and `sum_x_centered` accumulators
  - Pass 2: reads same data again to compute BN-backward epilogue using the sums
- Total CTAs: 40 on 148 SMs -- massive SM underutilization

The heuristic gap: `should_use_cooperative_reduction` returns False for `xhint=40 > 16`, and `reduction_split_factor` returns 1 because `reduction_numel=100352 < no_split_threshold=524288` on cc >= 10.0. So NEITHER the cooperative NOR the split path provides parallelism.

## Inductor Behavior (force_cooperative_reductions=True)

With `config.triton.force_cooperative_reductions = True`:
- Grid: `(RSPLIT, 40)` where RSPLIT = 3 (computed as `min(rnumel, last_power_of_2(148) // 40, MAX_RSPLIT)` = `min(100352, 128//40, 64)` = 3)
- Total CTAs: 40 * 3 = 120 on 148 SMs
- Same two-pass architecture (reduction + epilogue in one kernel)
- Uses `x_grid_barrier` for synchronization between splits
- Each CTA reduces ~33,450 elements

This provides 1.8x speedup over default but RSPLIT is severely limited by the grid barrier constraint (all CTAs must be simultaneously resident on SMs).

## Oracle Architecture

The oracle uses 3 separate kernels with no co-residency requirement:

1. **Partial dual-sum kernel** -- grid `(C=40, num_tiles=98)` = 3,920 blocks
   - Each block processes 1,024 elements of the reduction
   - Computes partial `sum_x` and `sum_x_centered` into a `[40, 98]` buffer
   - No grid barrier needed -- blocks are independent

2. **Finalize kernel** -- grid `(C=40,)` = 40 blocks
   - Combines 98 partials per channel
   - Also produces the `[40]` side-vector output

3. **BN epilogue kernel** -- grid `(ceil(4014080/256),)` = 15,680 blocks
   - Reads original inputs + reduction results
   - Writes channels-last `[512, 40, 14, 14]` output
   - Massively parallel pointwise

**Total blocks across all kernels: 19,640** vs Inductor's 120.

## Why the Remaining Gap Exists

| Factor | Inductor (cooperative) | Oracle |
|--------|----------------------|--------|
| Reduction parallelism | 120 CTAs (3 per channel) | 3,920 CTAs (98 per channel) |
| Epilogue parallelism | Same 120 CTAs (second loop) | 15,680 CTAs (dedicated kernel) |
| Grid barrier overhead | Yes (x_grid_barrier) | None (separate kernel launches) |
| Data read pattern | 2 passes through same data in same kernel | 2 passes in separate kernels with independent tuning |
| SM utilization | 120/148 = 81% (but only 3 splits of work) | 148/148 = 100% across many waves |

The fundamental architectural issue is that Inductor's `cooperative_reduction` uses grid barriers which require ALL CTAs to be simultaneously resident on the GPU. For `xnumel=40`, this limits RSPLIT to `floor(SM_count / xnumel) = 3`. The oracle avoids this by decomposing into separate kernel launches where synchronization is implicit via kernel boundaries.

## Scheduler Change Required

**Decomposed split-K reduction lowering** (distinct from existing `cooperative_reductions`):

1. **Detection**: When a reduction has `xnumel * RSPLIT_achievable < num_SM * 4` (i.e., grid-barrier cooperative cannot saturate the GPU), AND the reduction has dependent epilogue consumers that re-read the same inputs, decompose into separate kernels.

2. **Partial reduction kernel**: Grid `(xnumel, ceil(rnumel / BLOCK_K))`. Each block computes a partial accumulator. No grid barrier, no co-residency requirement. Store partials to a temporary buffer `[xnumel, num_tiles]`.

3. **Finalize kernel**: Grid `(xnumel,)`. Combines partials into final reduction results.

4. **Epilogue kernel**: Grid `(ceil(total_output_elements / BLOCK_ELEMS),)`. Uses the finalized reduction results to compute the dependent output.

This is essentially the pattern the oracle manually implements. The key insight is that for small `xnumel` with large `rnumel`, the grid-barrier approach is fundamentally bandwidth-limited because it cannot saturate the memory system with enough concurrent requests.

## Family Size

86 repros are classified as COOPERATIVE_SPLIT_K in the optimization queue. This fix would benefit the entire family.

## Fix Implemented

**Aggressive split factor for undersaturated GPUs** in `choices.py:reduction_split_factor`.

When `8 <= numel_hint < num_sm` (multi-output reductions that can't fill the GPU), reduce
`min_elements_per_thread` from 32 to 4, targeting 1024 elements per CTA (matching oracle patterns).
This targets ~8 waves of CTAs (num_sm * 8) capped by the work-per-CTA minimum.

For the target case (numel=40, rnumel=100352):
- Before: split=13, 520 CTAs (3.5 waves)
- After: split=30, 1200 CTAs (8.1 waves)

## Heuristic Gap Fix (Historical)

The prior gap had two components:
1. `should_use_cooperative_reduction` rejects `xhint > 16` (still true, but irrelevant -- split_reductions handles it)
2. `reduction_split_factor` returned too few splits for undersaturated GPUs (FIXED)

The `split_reductions_for_undersaturated_gpu` config (already default True) lowered the no_split_threshold
to 8192, enabling splitting. But the split factor itself was limited by `min_elements_per_thread=32` which
gives 8192 elements/CTA. The fix uses 4 elements/thread (1024 elements/CTA) for undersaturated cases.
