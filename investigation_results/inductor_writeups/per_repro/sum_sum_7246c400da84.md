## Compile: 352.13us, Oracle: 275.26us, Gap: 1.279x

## Diagnosis: SCHEDULER_FUSION

## Root cause: Inductor schedules the EfficientNet batch-norm-backward fragment as 4 separate kernels: (1) a pointwise kernel for the sigmoid/broadcast/reciprocal producer, (2) a reduction kernel for the sibling channel sums over [N,H,W]=[128,112,112], (3) a persistent reduction for finalizing channel statistics, and (4) a pointwise epilogue for the full [128,32,112,112] output tensor. The oracle computes this in 3 kernels by sharing the fused gradient producer across both channel reductions in one pass, finalizing per-channel sums, and using those finalized scalars to emit both the returned gradient tensor and the [32] side gradient vector. The key inefficiency is that Inductor materializes the large [128,32,112,112] producer intermediate to global memory before reading it back in the reduction pass.

## Fix path: Teach Inductor's scheduler to recognize that the pointwise producer feeds two compatible sibling channel reductions (sum over [0,2,3]) and fuse them into a single cooperative reduction kernel that also stores partial sums. Then fuse the epilogue pointwise (which depends on the finalized channel scalars) in a second pass that reads the same activation data. The main changes are in `torch/_inductor/scheduler.py` (score_fusion to recognize shared-producer sibling reductions) and the BN-backward full-scope template pattern.
