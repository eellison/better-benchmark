## Diagnosis

The oracle computes the complete Demucs slice-scatter plus sibling masked-sum scope by materializing the required zero-padded `[64,64,95696]` output and accumulating the `where(arg30_1, 0.0, getitem_3).sum(dim=[0,2])` result while reading the central f32 `[64,64,92844]` input once in a single fused Triton copy/reduce pass. Inductor currently emits three separate kernels that read the input data twice: once for the masked-sum reduction and once for the slice-scatter copy.

## Root cause

The repro has two sibling consumers of the same `[64,64,92844]` f32 input:
1. A `slice_scatter` into a zero-filled `[64,64,95696]` output (padding 1426 elements on each side of dim=2)
2. A `where(mask, 0.0, input).sum(dim=[0,2])` producing a `[64]` result

Inductor schedules these as independent work:
- **Kernel 0** (`triton_red_fused_full_sum_where_0`): An outer reduction with xnumel=1216 (64 channels x 19 splits) and r0_numel=312738, reading both the bool mask and f32 input to produce partial sums.
- **Kernel 1** (`triton_per_fused_full_sum_where_1`): A persistent reduction finalizer that reduces the 19 partials per channel to 64 final sums.
- **Kernel 2** (`triton_poi_fused_full_2`): A pointwise kernel over all 391,970,816 output elements that re-reads the f32 input and writes the zero-padded output.

The oracle reads the f32 input only once in a 3D grid `(CHANNELS=64, BATCH=64, TILES_T=23)` kernel that simultaneously:
- Copies the central slice to the padded output
- Applies the bool mask and accumulates partial sums

A separate small zero-pad kernel writes only the 2*1426*64*64 = ~11.6M padding elements (vs Inductor writing all 392M elements including redundant stores of the central data). The oracle's final-sum kernel reduces 64*23=1472 partials per channel.

The net effect: Inductor reads ~1.49 GB of input data (744 MB for kernel 0 + 744 MB for kernel 2), while the oracle reads ~744 MB total (one pass). This 2x bandwidth difference explains the 1.56x gap (not exactly 2x due to the write-side costs being equal).

## Kernel count

- Oracle: 3 kernels (_zero_pad_kernel + _copy_reduce_kernel + _final_sum_kernel), but the padding kernel is trivially small
- Inductor: 3 kernels (reduction + finalizer + pointwise copy), with the expensive reduction and copy both reading the full input

## Recommendation

Teach the Inductor scheduler to fuse a slice_scatter materialization with a sibling reduction over the same input tensor into a multi-output copy-plus-reduction kernel. The key insight is that when the scatter destination is zero-initialized and the source covers a contiguous slice, the copy and the reduction can share a single read of the source data. The zero-padding regions can be handled by a trivial separate kernel or folded into boundary logic.
