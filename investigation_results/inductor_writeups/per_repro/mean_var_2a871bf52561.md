## Pattern

BERT embedding assembly + LayerNorm: word embedding lookup (20005x768),
position embedding slice (1x512x768 -> 1x128x768), token type embedding (3x768),
sum the three embeddings, then compute LayerNorm (mean + var over hidden=768)
with affine (weight/bias). Three aliasing [16384, 768] views are returned.

## Kernel Count

- Oracle: 1 kernel (fully fused embedding lookups + layernorm in persistent mode)
- Inductor: 1 kernel (fused but uses looped reduction with R0_BLOCK < 768)

## Root Cause

Both produce 1 kernel. The difference is in codegen strategy:

1. **Persistent vs Looped**: The oracle uses BLOCK_H=1024 (next_power_of_2(768)),
   loading the full hidden dimension in one shot. Inductor's kernel uses a
   reduction loop (`for r0_offset in tl.range(0, r0_numel, R0_BLOCK)`) because
   its persistent threshold doesn't select persistent mode for rnumel=768 with
   this many embedding reads.

2. **Device assertions**: Inductor emits `tl.device_assert((0 <= tmp4) & (tmp4 < 20005), ...)`
   for each embedding index, adding overhead. The oracle skips bounds checks.

3. **Embedding gather efficiency**: The oracle uses direct pointer arithmetic
   for embedding lookups, while Inductor's indirect indexing with assertions
   adds instruction count.

The 1.4x gap is primarily from the looped reduction overhead (multiple passes
over the hidden dimension) vs the oracle's single-pass persistent approach, combined
with assertion overhead.

## Fix Direction

1. **Raise persistent threshold for gather-dominated reductions**: When a reduction
   kernel's primary compute is embedding gathers (indirect loads), the persistent
   threshold should be higher to avoid re-issuing the gathers in each loop iteration.
   File: `/tmp/pytorch-work/torch/_inductor/choices.py`

2. **Elide device_assert for provably-bounded indices**: The index tensors are
   generated by `Index(20005)` which guarantees bounds. The assertion is redundant.
   File: `/tmp/pytorch-work/torch/_inductor/lowering.py` (assert_indirect_indexing)
