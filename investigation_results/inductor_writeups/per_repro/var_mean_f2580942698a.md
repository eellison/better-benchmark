## Diagnosis

The oracle computes the complete Swin patch-merge residual add, 2x2 window-layout clone/view, population var_mean LayerNorm over the flattened patch channels, affine epilogue, and final contiguous output in one Triton row kernel. Inductor currently materializes the residual add and patch-merge clone/view before scheduling a generic var_mean LayerNorm and affine pointwise epilogue.

## Root cause

The repro performs a Swin Transformer patch-merging block:
1. `view(addmm, [B, H*W, C])` + residual add with `[B, H*W, C]` input
2. Reshape to `[B, H, W, C]`, then to `[B, grid_h, inner_h, grid_w, inner_w, C]`
3. Permute `[0,1,3,4,2,5]` + clone (gathers 2x2 patches into `[B, grid_h, grid_w, inner_h*inner_w*C]`)
4. `var_mean(..., [3], correction=0)` -- LayerNorm over hidden_out = inner_h*inner_w*C
5. Affine epilogue: `(x - mean) * invstd * weight + bias`
6. View to final `[B*grid_h*grid_w, hidden_out]`

The materialized clone at step 3 requires reading the residual-added result and writing it in a new layout before the LayerNorm can proceed. The oracle fuses steps 1-6 by mapping each output row directly to its source spatial positions using the patch-merge index formula, computing the residual add on-the-fly, normalizing, and storing the final result -- eliminating the intermediate clone buffer entirely.
