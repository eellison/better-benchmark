## Diagnosis

The oracle computes the complete DeepRecommender SELU-style pointwise producer once, writes the returned f32[197951,1024] permute-view backing storage, and accumulates the sibling f32[197951] column sum in the same Triton kernel using Inductor-matching f32 libdevice exp. Inductor currently schedules the required permuted side output and the column reduction as separate generic work over the same expression.

## Root cause

The repro computes a SELU-like pointwise expression over two `[1024, 197951]` inputs:
```
value = where(arg17 <= 0, arg18 * 1.758 * exp(arg17), arg18 * 1.0507)
```
Then it produces two outputs:
1. A `[197951, 1024]` permuted view (transpose) of the computed values
2. A `[197951]` column-wise sum (reduction over dim=0 of the `[1024, 197951]` result)

Inductor cannot fuse the permuted materialization (which needs to store `[1024, 197951]` in column-major order) with the column reduction (which reduces over the 1024 rows for each of the 197951 columns) because a permuted layout store and a sibling reduction over the same producer expression require a multi-output kernel pattern that is not supported today.

The oracle iterates over columns (BLOCK_N columns per CTA), loops over all 1024 rows for those columns, stores the computed values in row-major order, and accumulates the column sum -- achieving one read of inputs and one compute of the SELU expression per element.

## Recommendation

Teach the scheduler/codegen to emit a multi-output pointwise-plus-column-reduction kernel when a required permuted materialization and a reduction consume the same pointwise expression. The key insight is that when iteration is column-major (to service the reduction), the store to the backing buffer can use the same column offset.
