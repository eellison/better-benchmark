# sum_sum_ba803596a087

## Compile: 3342us, Oracle: 1833us, Gap: 1.82x

## Diagnosis: COMPREHENSIVE_PADDING_STRIDE_MISMATCH

## Classification: PERMUTE_SIDE_SIBLING

The graph has two outputs from the same intermediate (add_tensor [256, 128, 30522]):
1. `sum_dim_int_list_1`: column reduction to [30522] (output 0)
2. `constant_pad_nd([0,2,0,0])`: right-pad to [32768, 30524] (output 1)

## Root Cause (deep investigation)

The "second output" (`constant_pad_nd_default`) is a right-padding of the intermediate `add_tensor`
by 2 columns. It produces [32768, 30524] stride [30524, 1] (contiguous).

Inductor's fused reduction kernel already optimally:
- Reads inputs (arg4_1 [256,128,30522] + arg3_1 [32768,30522] + labels + scalars)
- Computes add_tensor inline (no separate kernel)
- Stores add_tensor values to buf0 as a side-output
- Simultaneously accumulates the row reduction (sum over 32768 rows -> [30522])

**The problem is the STRIDE MISMATCH between buf0 and the padded output:**
- buf0 gets stride [30528, 1] due to `comprehensive_padding` (128-byte row alignment: `ceildiv(30522,32)*32 = 30528`)
- The padded graph output requires stride [30524, 1] (contiguous [32768, 30524])
- Since 30528 != 30524, a full-tensor COPY kernel is needed (reads 4 GB, writes 4 GB = 8 GB extra traffic)

**Memory traffic breakdown:**
- Kernel 0 (zero-fill): 65536 elements (trivial, fills last 2 columns of buf4 with zeros)
- Kernel 1 (fused reduction): ~12 GB (reads 8 GB inputs, writes 4 GB buf0 + negligible reduction output)
- Kernel 2 (COPY): ~8 GB (reads 4 GB from buf0 stride 30528, writes 4 GB to buf4 stride 30524)
- **Total: ~20 GB**

**Oracle total: ~12 GB** (writes directly to output with stride 30524, no separate copy)

## What the Oracle Does Differently

The oracle uses 3 custom Triton kernels:
1. `_store_and_partial_sum_kernel`: tiles over (vocab_blocks x row_blocks), for each tile:
   - Computes add_tensor values
   - Stores directly to `side_out_ptr + row*PADDED_VOCAB + col` (stride = PADDED_VOCAB = 30524)
   - Accumulates partial row-sums
2. `_reduce_partials_stage1_kernel`: reduces partial sums across row blocks
3. `_reduce_partials_stage2_kernel`: final reduction

Key difference: the oracle stores the intermediate with stride PADDED_VOCAB (30524) directly
into the output buffer, avoiding the copy. Inductor stores with stride 30528 (alignment)
and then copies to stride 30524.

## Why Can't We Fix This?

### Attempted approaches:

1. **inplace_constant_pad_nd**: This optimization would reuse the buffer's padding as the
   constant_pad_nd output. BUT:
   - It calls `x.realize()` which breaks the fusion between the computation and the reduction
   - The input is a `ReinterpretView` (from `view_default_4`), not a direct `StorageBox`
   - Even when extended to handle ReinterpretView, the realize() breaks the optimal fused kernel

2. **ConcatKernel stride matching**: Making ConcatKernel adopt the input's stride (30528)
   eliminates the concat-internal copy, but the graph output REQUIRES stride [30524, 1]
   (from FX metadata), causing another copy from [30528,1] to [30524,1].

3. **Suppress comprehensive_padding**: Setting stride to 30522 (natural) means stride < 30524,
   so inplace padding can't work. Setting stride to 30524 exactly requires custom logic
   that knows about the downstream pad amount at layout-freeze time.

3. **Preferred stride annotation (IMPLEMENTED, still fails)**: A pre-lowering pass
   successfully identifies the pattern and annotates `preferred_inner_strides = {30522: 30524}`.
   Modifying `_pad_strides()` to use 30524 instead of 30528 works -- the buffer IS allocated
   with stride [30524, 1]. BUT the copy still happens because of a DEEPER ordering issue:

4. **The ordering problem (fundamental)**: The reduction (`sum_dim_int_list_1`) is lowered
   BEFORE the `constant_pad_nd_default` in topological order. When the reduction calls
   `make_loader()` on add_tensor, it triggers `mark_reuse()` -> `realize()`, which creates
   a ComputedBuffer. The ConcatKernel (from _pad_as_cat) runs AFTER this, and finds the
   buffer already realized. Even with matching strides, `realize_into` cannot redirect an
   already-realized buffer into the concat output -- it creates a copy Pointwise instead.

   The fix would require the scheduler to recognize that the realized buffer and the
   ConcatKernel destination share the same stride, and directly alias them (output planning).

### The Design Gap:

The core issue is a REALIZATION ORDERING problem:
1. The reduction realizes add_tensor into a separate buffer (buf0)
2. ConcatKernel allocates a separate output (buf4) for the padded graph output
3. Even with matching strides (both [30524, 1]), these are different allocations
4. A copy is needed to move data from buf0 to buf4[:, :30522]

The oracle avoids this by computing directly into the padded output buffer -- a
"write-through" approach where the kernel's store target IS the graph output.

**Fix direction (scheduler-level)**: The scheduler needs "output planning" -- when a
realized buffer's ONLY other consumer (besides the reduction that produced it) is a
ConcatKernel slice with matching stride, the buffer should be allocated as a
NonOwningLayout pointing into the concat output. This requires:
- Detecting the pattern before the reduction triggers realization
- Pre-allocating the concat output and setting up NonOwningLayout on the intermediate
- This is essentially what ConcatKernel.realize_into does for UNREALIZED buffers
  (FlexibleLayout), but it fails for already-realized buffers (FixedLayout)
