## Operation
Takes `[384, 64, 512]` f32 attention BMM output:
1. Views as `[32, 12, 64, 512]`, scales by 0.3535 (1/sqrt(head_dim))
2. Permutes to `[32, 12, 512, 64]` then `[32, 512, 12, 64]`
3. Clones to contiguous `[32, 512, 768]` -> views as `[16384, 768]`
4. Produces two outputs:
   - `permute_default_2`: `[768, 16384]` transposed view (non-materialized permute)
   - `view_default_3`: `[768]` column sum (reduction over dim 0)

## Kernel Count
- Inductor: 3 kernels (layout clone + two reductions)
- Oracle: 1 kernel (fused scale + layout store + reduction)

## Root Cause Analysis

Inductor generates 3 kernels:
1. **triton_poi_0** (2D tiled, ynumel=768, xnumel=16384): Computes scale*permute and writes the contiguous `[16384, 768]` backing storage (the clone for the transposed view)
2. **triton_red_1** (xnumel=768, r0_numel=16384): Reads the clone output and reduces each column to produce the `[768]` sum. This is a separate kernel because it's a reduction.
3. **triton_red_2**: Another reduction pass (likely a multi-stage reduction for large r dimension)

The oracle fuses everything into a single kernel that:
- Iterates over the source `[384, 64, 512]` tensor once
- For each element, computes the target position in the `[16384, 768]` layout
- Writes the scaled value to the backing store (output 0)
- Accumulates into a per-column accumulator (output 1)

**Key inefficiency**: The `[16384, 768]` intermediate is written by kernel 0 and then re-read by kernels 1+2 for the reduction. This is ~48 MB of extra memory traffic (write + read) that the oracle avoids by accumulating during the layout store.

## Why Inductor Cannot Do This Today

1. **Reduction cannot be fused with layout-changing pointwise store**: Inductor's scheduler separates the contiguous clone (pointwise kernel writing to new layout) from the column-sum reduction because they have different iteration patterns — the clone iterates over output elements while the reduction iterates over reduction groups.

2. **Multi-output with mixed pointwise + reduction not supported**: The scheduler cannot emit a kernel that has both a direct store epilogue (for the clone backing) and a reduction epilogue (for the sum). This would require a "pointwise-with-accumulator" template.

3. **The transposed permute blocks fusion**: The `permute([1, 0])` on the output means output 0 is a non-contiguous view of the clone. The scheduler must produce the contiguous backing storage, and this store has a different memory access pattern than the reduction.

## Fix Direction

**SCHEDULER_FUSION**: Add a "reduction-with-materialization" kernel template:
- File: `/tmp/pytorch-work/torch/_inductor/scheduler.py`
- When a pointwise kernel producing a layout materialization (clone) has a downstream reduction consumer over the same data, fuse them into a single kernel that stores the materialization AND accumulates the reduction simultaneously.

This is a common pattern in transformer backward passes where attention output must be both stored in a new layout (for downstream GEMM) and reduced (for bias gradient).

## Affected Repros
This pattern likely appears in all BERT/GPT attention backward passes where the attention output feeds both a linear layer (requiring layout materialization) and a bias gradient sum.
