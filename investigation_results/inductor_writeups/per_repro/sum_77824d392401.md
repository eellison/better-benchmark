## Root Cause

The repro (from genai CrossEntropyBackward) computes the dense cross-entropy backward gradient over [8192, 262144]:
- Labels: int64[8192]
- Logits: bf16[8192, 262144]
- Row shifts: f32[8192, 1] (two sets)
- Output: bf16[8192, 262144]

### Oracle approach (algebraic elimination):
The oracle recognizes that the one-hot row reduction `sum(eq(iota, label) * ...)` over 262,144 columns is algebraically equivalent to a single indexed scalar lookup per row. Instead of scanning the full row to compute the sum, it directly computes `row_sum = where(valid_label, -scale, 0.0)` and `sparse = where(col == label & valid, -scale, 0.0)`. The kernel then computes the dense exponential epilogue `exp(logit - shift0 - shift1) * row_sum` and adds the sparse correction, all in one pass.

### Inductor approach:
Inductor materializes the full [8192, 262144] equality expansion, scans each 262K-wide row to compute the one-hot sum reduction, then rereads the same row for the exponential backward epilogue. This doubles the memory traffic for rows.

### Why the oracle is 1.21x faster:
1. **Eliminates redundant scan**: The one-hot sum is O(1) per row (just check the label), not O(V) where V=262144.
2. **Single-pass output**: The oracle writes the output in one pass combining the sparse label correction and dense exponential term, while Inductor needs a reduction pass followed by an epilogue pass.
3. **Reduced memory traffic**: No need to materialize or read back the one-hot expansion.
