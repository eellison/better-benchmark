## Scope

The repro computes the MobileViT MLP activation-gradient pointwise producer

```text
z = mm_66 * sigmoid(addmm_2) * (addmm_2 * (1 - sigmoid(addmm_2)) + 1)
```

and returns both:

- `z.reshape([131072, 288]).permute([1, 0])`, shape `[288, 131072]`, stride `(1, 288)`
- `z.sum(dim=0)`, shape `[288]`, stride `(1,)`

The oracle covers this full compiled `repro.py` scope with the same inputs,
outputs, dtypes, shapes, and strides. It writes the backing storage for the
transposed side output while accumulating per-feature partial sums in Triton,
then finalizes the `[288]` sum with a second Triton kernel. Torch is used only
for allocation, views, reference checking, and compile baselines.

## Gap Diagnosis

The oracle differs from a naive schedule by avoiding a separate full read of the
materialized pointwise producer for the feature reduction. In local measurement
that is faster than the required current-branch compile configs, but it is still
slower than the historical best compile timing from the queue:

- Oracle, full scope: `145.120 us`
- `coordinate_descent_tuning=True`: `177.376 us`
- `combo_kernels=True,combo_kernel_per_subkernel_blocks=True,coordinate_descent_tuning=True,benchmark_combo_kernel=True,triton.multi_kernel=3`: `175.200 us`
- Historical `best_compile_us`: `87.90399879217148 us`

Because the full-scope Triton oracle does not beat the historical-best gate,
this is not a true floor. The result is consistent with the historical compile
already reaching the materialization-plus-reduction throughput floor for this
shape; no missing Inductor optimization should be claimed from this artifact.

## Inductor Fix

No Inductor change is justified by this measurement. If a future full-scope
oracle beats the historical gate, the relevant compiler direction would be a
`SCHEDULER_FUSION` template that writes a required layout side output and
accumulates compatible side reductions from the same producer pass. This
artifact should remain diagnosis-only unless that gate is cleared.
