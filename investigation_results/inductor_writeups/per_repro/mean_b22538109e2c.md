# mean_b22538109e2c - NFNet GELU Spatial Mean

## Benchmark Results
- Oracle: 27.42 us
- Compiled: 36.35 us
- Ratio: 1.326x (oracle wins)

## Classification
SCHEDULER_FUSION - expensive pointwise producer not optimally sunk into spatial mean reduction

## Root Cause

The oracle computes the full NFNet exact-erf GELU, fp16 rounding, gamma scale (1.7015), and 6x6 spatial mean in one Triton reduction kernel, producing the final fp16 [128, 3072] view directly.

The computation on input f16[128, 3072, 6, 6]:
1. Convert to f32
2. GELU: x * 0.5 * (1 + erf(x * 0.7071))
3. Convert back to f16
4. Scale by 1.7015 (NFNet gamma)
5. Spatial mean over 6x6 = 36 elements
6. View to [128, 3072]

Inductor generates a single fused kernel (`triton_per_fused_add_convert_element_type_erf_mean_mul_0` with xnumel=393216, r0_numel=36) that handles all operations. With 128*3072=393216 rows each reducing 36 elements, this is already fused.

The gap (1.326x) is significant. Looking at the Inductor kernel shape: xnumel=393216, r0_numel=36. This is a persistent reduction over only 36 elements per row. The issue may be that:
1. The oracle uses a coarser tiling (batch*channel_block) that processes multiple channels per program, improving instruction-level parallelism
2. Inductor assigns one program per row (393216 programs total), which creates excessive grid overhead for this many tiny reductions
3. The erf function is expensive and with 393216 independent rows, the oracle may batch multiple rows per block for better utilization

## Kernel Count
- Oracle: 1 kernel (GELU + fp16 + scale + spatial mean, multi-row tiled)
- Inductor: 1 kernel (same fusion, row-per-program)

## Config Exploration
- `combo_kernels = True`: no effect (already one kernel)
- `coordinate_descent_tuning = True`: may tune XBLOCK but cannot change fundamental tiling
- The 1.326x gap suggests a tiling/occupancy issue rather than algebraic

## Why Inductor Cannot Do This Today

The issue is that Inductor's persistent reduction for very small r0_numel (36) with very large xnumel (393216) creates too many programs with too little work each. The oracle likely:
1. Tiles multiple rows per block (e.g., XBLOCK=4 rows, each reducing 36 elements)
2. This amortizes the erf computation overhead and kernel scheduling across multiple rows
3. With 36 elements per row, the reduction fits trivially in registers, so the bottleneck is the per-row pointwise compute (erf)

Inductor's codegen for persistent reductions with small r0_numel may not optimally choose XBLOCK to batch enough rows together, leading to underutilized warps computing expensive erf operations.

## Design Doc

**Fix location**: `/tmp/pytorch-work/torch/_inductor/codegen/triton.py` and `/tmp/pytorch-work/torch/_inductor/choices.py`

**Enhancement needed**: For persistent reductions with very small reduction dim (r0_numel <= 64) and expensive per-element ops (erf, exp), increase XBLOCK to process multiple output rows per program. The heuristic should account for:
- Per-element compute cost (erf is ~10x more expensive than add)
- Register pressure from retaining the tile
- Optimal occupancy given the large xnumel

**Affected repro count**: NFNet uses exact-erf GELU extensively. This specific pattern (GELU + spatial mean with small HW) likely appears in 2-3 repros. The general issue of small-reduction-dim tiling affects many spatial mean patterns.
