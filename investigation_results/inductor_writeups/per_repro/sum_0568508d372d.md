## Root Cause

The oracle computes the masked matrix return ([4096, 1024] transposed-view layout) and sibling column-sum return ([4096]) from the same tile load in one Triton reduction kernel. It loads each tile once, applies the `arg <= 0` zero mask, writes the required transposed layout, and accumulates the sum side output.

Inductor schedules the decomposed where, permute metadata, and dim-0 sum through generic reduction scheduling with extra multi-output reduction overhead. The scheduler does not form a full-scope plan that shares the masked pointwise producer between a materialized layout-changing output and a sibling column reduction.

## Kernel count
- Oracle: 1 kernel (fused masked transpose + column sum)
- Inductor: 2+ kernels (layout materialization + separate reduction)

## Recommendation

Requires scheduler enhancement: teach reduction scheduling to keep same-source pointwise producers virtual and emit one multi-output reduction/materialization kernel for compatible static shapes where one output is a layout-changing store and another is a sibling reduction.
