## Operation

Takes `f32[2048, 512]` arg, scalar `f32[]` full, `f32[2048, 512]` mm, computes:
```
where_self = where(arg <= 0, full, mm)   # f32[2048, 512]
permute = permute(where_self, [1, 0])    # f32[512, 2048] (transposed view)
sum = sum(where_self, dim=[0])           # f32[512]
return (permute, sum)
```

## Root Cause

The oracle fuses the pointwise `where` computation with both:
1. Writing the result into transpose-storage layout (backing the returned permute view)
2. Accumulating the column-wise sum (dim=0 reduction)

...all in a single Triton kernel that tiles over columns with full-row BLOCK_ROWS=2048.

Inductor generates 3 kernels:
1. `triton_poi_fused_le_where_0`: pointwise kernel computing `where` and materializing to a contiguous buffer
2. `triton_red_fused_sum_1`: outer reduction reading that buffer with OUTER reduction hint (8192 x-elements, 128 r-elements)
3. `triton_per_fused_sum_2`: persistent reduction finalizing the sum partials

The extra cost comes from:
- One full write + read round-trip of the 2048x512 = 4MB buffer (write in kernel 1, read in kernel 2)
- The permute output is a zero-cost view of that buffer, but the sum cannot be fused with the pointwise because Inductor's scheduler does not support emitting a single kernel that both materializes a layout-changing output and performs a sibling reduction over the same producer

## Why Inductor Cannot Do This Today

The scheduler cannot fuse a required materialized output (the `permute` view which forces the pointwise to be realized with specific strides) with a sibling reduction consumer that operates on the same expression. When `permute_default` is a graph output, Inductor must realize `where_self` to backing storage. The sum then becomes a separate reduction over that realized buffer.

The missing capability is a multi-output kernel template that can simultaneously:
- Write computed values to a buffer with arbitrary output strides (for the permute view)
- Accumulate a reduction over those same values in registers

## Design Doc

**Enhancement needed**: Teach the scheduler to recognize the pattern where a pointwise producer has both:
1. A required materialization (graph output via permute/view)
2. A sibling reduction consumer (sum over same dims)

And emit a single kernel that writes the materialized output and accumulates the reduction simultaneously. This is the "multi-output pointwise-plus-column-reduction" template pattern.

**Affected repros**: sum_cb19b2c26400, sum_d657e5a1c5df (and likely other DLRM/DeepRecommender patterns with permute+sum siblings).
