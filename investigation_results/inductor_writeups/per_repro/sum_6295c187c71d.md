## Root Cause

The oracle computes the DeBERTa reshape/permute/clone/divide scope by materializing the required contiguous [4096,1536] base buffer and accumulating the sibling [1536] column sum from the same scaled layout pass. Inductor schedules the required layout-producing clone and the reduction as generic work around the materialized buffer.

The scheduler does not fuse a required materialized layout side output with a downstream reduction consumer that reads the same logical tensor through view/permute structure.

## Kernel count
- Oracle: 1 kernel (fused layout copy + partial column sum)
- Inductor: 2+ kernels (layout materialization + separate column reduction)
