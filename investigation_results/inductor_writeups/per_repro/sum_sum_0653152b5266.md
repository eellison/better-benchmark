## Root Cause

The oracle exploits the structured Longformer scatter pattern: it knows the
`_low_memory_max_pool_offsets_to_indices`-like scatter coordinates are deterministic
from the block/chunk/window/head_dim structure, so it emits the scatter via
`atomic_add` directly from the source layout into the `[768, 2048]` side outputs
(column-major), then reduces those layouts in a single pass per hidden dim.

Inductor instead materializes the intermediate index/scatter buffers via generic
`index_put(accumulate=True)` lowering, producing multiple intermediate kernels:
1. A pointwise kernel to generate the iota indices
2. A clone+index_put kernel for the actual scatter
3. A layout copy for the strided output
4. Two reduction kernels for the sibling sums

The fundamental issue is that Inductor's `index_put(accumulate=True)` lowering
does not recognize structured scatter patterns (where indices are computed from
known strides/block structure) that could be replaced by a fused scatter-reduce
without materializing intermediate buffers.

## Design Doc

**Why Inductor cannot do this today**: The scheduler sees `index_put(accumulate=True)` 
as an opaque scatter operation that must materialize its result before downstream
consumers (the `as_strided` views and reductions) can proceed. It cannot recognize
that the index tensor has deterministic structure from the Longformer attention
pattern (block*chunk*window*head_dim -> flat offset) and that the scatter + layout +
reduction sequence could be replaced by direct structured writes.

**What enhancement is needed**: A structured scatter-reduce FX pass in
`torch/_inductor/fx_passes/` that:
1. Detects `full -> index_put(accumulate=True) -> as_strided -> sum` patterns
2. Analyzes the index tensor's derivation (iota + as_strided) to extract structure
3. Replaces the pattern with a fused scatter-reduce template that avoids buffer materialization

**Affected files**: `torch/_inductor/fx_passes/post_grad.py`, potentially new pass file
