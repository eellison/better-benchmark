# sum_51d2ed69e698 — MULTI_OUTPUT_SHARED_REDUCTION + BROADCAST_TRANSCENDENTAL_RECOMPUTATION

## Summary
- **Model**: vllm_mistralai_Mistral-7B-Instruct-v0.3
- **Pattern**: grouped-head sum + RoPE half-rotation + transposed output layout
- **Ratio**: 2.19x -> 1.27x (`realize_heavy_transcendentals_on_reuse`) -> **~1.00x AT_FLOOR** (`rotate_half_gather`)
- **Status**: CLOSED — two fixes: `config.realize_heavy_transcendentals_on_reuse`
  (commit `b1e8a30af15`) + `config.rotate_half_gather` (commit `1406552b9d3`)

## Operation
Takes two `[4, 32, 512, 128]` bf16 inputs (getitem_1, getitem) plus `[64]` f32 frequency table:
1. Views getitem_1 as `[4, 8, 4, 512, 128]`, sums over dim 2 (groups of 4 heads)
2. Computes rotary table: sin/cos from iota*freqs, expanded to `[1, 512, 128]`
3. Applies RoPE half-rotation to the grouped sum (slice_scatter neg pattern)
4. Applies same RoPE half-rotation to getitem (full 32-head path)
5. Both outputs are permuted+cloned to transposed `[C, T]` layout

## Kernel Count
- Oracle: 3 kernels (table gen 65K + grouped_rope 2M + full_rope 8M)
- Inductor (before): 3 kernels (sum*sin 2M + grouped_rotation 2M + full_rope 8M)
- Inductor (after fix): 4 kernels (sin/cos table 65K + sum*sin 2M + grouped_rot 2M + full_rope 8M)

## Benchmark
- Oracle: 20.2 us
- Compiled (before): 44.5 us (ratio 2.19x)
- Compiled (after fix): 25.6 us (ratio **1.27x**)

## Root Cause Analysis

### Primary bottleneck: Broadcast transcendental recomputation (FIXED)

The sin/cos frequency table is `[1, 512, 128]` (65K unique values) but gets consumed by:
- `[4, 8, 512, 128]` grouped RoPE kernel (32x broadcast)
- `[4, 32, 512, 128]` full RoPE kernel (**128x broadcast**)

Without the fix, every thread in the 8M-element full RoPE kernel recomputes
`sin(freq[dim%64] * seq_pos)` and `cos(freq[dim%64] * seq_pos)` — transcendentals
that cost ~10-20 clock cycles each. With the fix, a tiny 65K-element kernel
precomputes the sin/cos tables (~128KB), and the 8M-element kernel just loads from
them (1-2 cycles per element).

**Profiled per-kernel impact:**
| Kernel | Before fix | After fix | Speedup |
|--------|-----------|-----------|---------|
| Full RoPE (8M) | 32.1 us | 9.6 us | **3.3x** |
| Grouped path | 11.1 us | 8.1 us | 1.4x |
| Table gen (new) | N/A | 2.0 us | - |

### Secondary (unfixed): Shared reduction recomputation

The grouped sum `squeeze_dim` has 2 users (`sum*sin` and `sum*cos`). With
4 reads and the threshold at >4, the sum is NOT materialized. Kernel 1 must
re-read 4 source heads to recompute the sum for the cos path.

Materializing the sum separately was tested but is **not profitable** — the extra
kernel launch overhead (~5us) outweighs the saved reads because the sum's source
data (arg0_1) benefits from cache locality when re-read within the same kernel.

### Tertiary (unfixed): slice_scatter rotation pattern

The RoPE half-rotation generates branch-heavy code with conditional loads
(`tl.where`), whereas the oracle uses a clean unconditional gather with sign flip.
This adds ~1-2us per kernel but is architecturally hard to fix without an FX pass
that pattern-matches the slice_scatter rotation.

## Fix Implemented

**Commit**: `b1e8a30af15` on `pr-184905`
**Config**: `torch._inductor.config.realize_heavy_transcendentals_on_reuse = True` (default enabled)
**Files changed**:
- `/tmp/pytorch-work/torch/_inductor/config.py` (new flag)
- `/tmp/pytorch-work/torch/_inductor/ir.py` (`should_realize_on_reuse`)

**Mechanism**: When a multi-user Pointwise node contains expensive transcendental
ops (sin, cos, tan, asin, acos, atan, atan2), force materialization. This ensures
the expensive computation happens once and downstream consumers cheaply load from
the precomputed buffer.

## Remaining Gap — CLOSED by rotate_half_gather (2026-06-09)

**Commit**: `1406552b9d3` on `pr-184905`
**Config**: `torch._inductor.config.rotate_half_gather = True` (default enabled,
env `TORCHINDUCTOR_ROTATE_HALF_GATHER`)
**Files**: new `torch/_inductor/fx_passes/rotate_half_gather.py`, registration in
`post_grad.py`, flag in `config.py`.

A post-grad FX pass recognizes the RoPE half-rotation —
`add(slice_scatter(zeros, neg(x[..., H:]), hi), slice_scatter(zeros, x[..., :H], lo))`
and the `cat([x2, -x1])` form — and rewrites it to a single-load gather:

    out[..., j] = sign(j) * src[..., where(j < D-H, j+H, j-(D-H))]

built from `iota + where + _unsafe_index + (2*cond-1) sign multiply`. The
where-on-iota index folds to register-only index arithmetic in Triton
(`tl.where(x0 < 64, 64+x0, -64+x0)` feeding ONE `tl.load`), eliminating the
dual masked loads from the slice_scatter lowering (`lowering.py:slice_scatter`,
the `ops.masked(mask, src_load, 0) + where` at torch/_inductor/lowering.py
~L3975-4015) and from `pointwise_cat` (~L1694). The sign is applied as a
multiply (not `where(cond, g, -g)`) so the gather stays single-use —
otherwise `realize_indirect_indexing_on_reuse` realizes it into a separate
kernel.

**Benchmark (B200, CUDAGraph, interleaved A/B, fresh cache):**
| Config | compile_us | ratio |
|--------|-----------|-------|
| OFF (baseline w/ transcendental fix) | 24.7-25.4 | 1.24-1.27x |
| ON | 18.0-18.1 | **0.995-1.014x (AT_FLOOR)** |

Note: the gather formulation also lets the scheduler keep 3 kernels and the
"extra kernel launch" + "branch" overheads disappear together.

**Regression spot-checks (OFF -> ON, all correctness PASS):**
- sum_096e9f1570b5 (grouped RoPE): 10.94 -> 11.01 us (neutral)
- sum_a3c9eb6b27f1 (RoPE scatter reduce): 18.02 -> 13.76 us (**beats oracle**, 0.82x)
- pointwise_2cb944a69993 (cat-form GPT-NeoX RoPE, fires): 8.80 -> 8.99 us (neutral)
- pointwise_750904e8518d (GPT-J interleaved RoPE, no match): neutral
- pointwise_88280aa05d1d (generic qkv cat, no match): 46.02 -> 45.92 us
- pointwise_27183a793fcd (stencil slice_scatter, no match): 37.7 -> 36.6 us
- Edge cases verified: unequal halves, no-sign / both-negative rotations,
  two-tensor cat (no rewrite), overlapping slices (no rewrite), dynamic shapes
  (skipped safely).
- Pre-existing failure `test_slice_scatter_reinplace_cuda` (kernel-count
  assertion) fails identically on the clean branch — unrelated.

## Affected Repros
This pattern (broadcast transcendental recomputation) likely affects all RoPE-based
models (Mistral, Llama, GPT-NeoX families) where sin/cos position embeddings are
computed from frequency parameters and broadcast across batch/heads.
