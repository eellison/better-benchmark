# sum_51d2ed69e698 — SCHEDULER_FUSION

## Summary
- **Model**: vllm_mistralai_Mistral-7B-Instruct-v0.3
- **Pattern**: grouped-head sum + RoPE half-rotation + transposed output layout
- **Ratio**: 2.187x
- **Status**: Significant gap — scheduler fusion opportunity

## Operation
Takes two `[4, 32, 512, 128]` bf16 inputs (getitem_1, getitem) plus `[64]` f32 frequency table:
1. Views getitem_1 as `[4, 8, 4, 512, 128]`, sums over dim 2 (groups of 4 heads)
2. Computes rotary table: sin/cos from iota*freqs, expanded to `[1, 512, 128]`
3. Applies RoPE half-rotation to the grouped sum (slice_scatter neg pattern)
4. Applies same RoPE half-rotation to getitem (full 32-head path)
5. Both outputs are permuted+cloned to transposed `[C, T]` layout

## Kernel Count
- Inductor: 3 kernels
- Oracle: 3 kernels (table gen + grouped_rope + full_rope)

## Benchmark
- Oracle: 20.32 us
- Compiled: 44.45 us
- Ratio: 2.187x

## Root Cause Analysis

The oracle uses 3 specialized kernels:
1. **_rotary_table_kernel** (65536 elements): computes sin/cos table from frequency parameters
2. **_grouped_rope_kernel** (2097152 elements): reads 4 heads, sums them inline, applies RoPE, writes `[T, G*D]` output
3. **_full_rope_kernel** (8388608 elements): reads 1 head, applies RoPE, writes `[T, H*D]` output

Inductor generates 3 kernels but with critical inefficiency:
1. **Kernel 0** (2097152 elements, 5 loads + 1 store): Computes grouped sum and multiplies by sin table -> writes intermediate `mul_2`
2. **Kernel 1** (2097152 elements, 7 loads + 1 store): Reads intermediate `mul_2` AND re-reads 4 source heads to re-sum them (for the cos path), applies slice_scatter rotation pattern, writes grouped output with permute
3. **Kernel 2** (8388608 elements, 6 loads + 1 store): Full RoPE for all 32 heads, also with slice_scatter rotation

**Key problem**: Kernel 1 re-reads all 4 source heads (4 loads of 65536 stride) to recompute the sum for the cos multiplication path. The sum was already computed in Kernel 0 but only the `sum * sin` intermediate was stored — not the raw sum itself. This doubles memory traffic for the grouped path.

**Secondary problem**: The slice_scatter rotation pattern (`full -> slice_scatter(neg(slice(0:64)), 64:128) -> slice_scatter(slice(64:128), 0:64) -> add`) generates branch-heavy code with conditional loads, whereas the oracle directly computes `src_dim = dim < 64 ? dim+64 : dim-64` for a single unconditional gather.

## Why Inductor Cannot Do This Today

1. **Scheduler does not share reduction results across multiple consumer kernels**: The grouped sum `squeeze_dim` is used by both the sin-multiply path and the cos-multiply path. Inductor's scheduler materializes `sum * sin` (kernel 0) but then must re-derive the sum in kernel 1 for `sum * cos`.

2. **No multi-output kernel with shared reduction**: The ideal code would compute the sum once and apply both sin and cos rotations in a single kernel (as the oracle's `_grouped_rope_kernel` does). This requires a "reduction + multi-output pointwise epilogue" fusion pattern.

3. **slice_scatter rotation lowering is suboptimal**: The `full -> slice_scatter -> slice_scatter -> add` pattern for RoPE half-rotation could be algebraically simplified to a direct gather with sign flip, eliminating branches and the zero-fill tensor.

## Fix Direction

**Primary fix (SCHEDULER_FUSION)**: The scheduler needs to recognize that when a reduction result feeds two pointwise consumers that have the same iteration space, they should be fused into a single kernel with the reduction computed once. Specifically:
- File: `/tmp/pytorch-work/torch/_inductor/scheduler.py`
- Enhancement: Allow `can_fuse` to merge pointwise nodes that share a common reduction producer, emitting the reduction once with multiple store epilogues.

**Secondary fix (FX_PASSES)**: Add an algebraic rewrite in `torch/_inductor/fx_passes/post_grad.py` that simplifies the RoPE slice_scatter pattern:
```
full(0) -> slice_scatter(neg(slice(x, dim, 0, half)), dim, half, end) 
         -> slice_scatter(slice(x, dim, half, end), dim, 0, half) -> add
```
into a direct half-rotation gather: `x[..., half:] * sin - x[..., :half] * sin` (for the negative half) and the corresponding cos term.

## Affected Repros
This pattern appears in all Mistral/Llama-family RoPE backward passes with grouped-query attention (GQA). Similar repros: sum_51d2ed69e698 and likely other `oracle_rope_scatter_reduce` variants.

## Design Doc

The core scheduler enhancement needed:
1. Detect when a `ReductionNode` has multiple downstream `PointwiseNode` consumers with compatible iteration spaces
2. Fuse them into a single kernel that computes the reduction once and writes multiple outputs
3. This is related to but distinct from combo_kernels (which handles independent kernels); this is about shared-producer fusion

This would eliminate the redundant 4-head re-read (saving ~4MB memory traffic) and reduce kernel count from 3 to 2 for the grouped path.
